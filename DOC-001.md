# 1. SYSTEM ARCHITECTURE

## 1.1 High-Level Architecture
The Zayjar Restaurant SaaS Platform is built using a modern, multi-tenant cloud-native architecture designed to ensure maximum isolation, operational resilience, and sub-second execution speeds. The system uses a logical domain separation model (modular monolith) implemented in **NestJS (Node.js)** for the backend layer, coupled with high-availability relational databases and containerized background worker clusters.

### Infrastructure Core Services:
- **DNS & Edge Routing (Route 53):** Handles dynamic tenant mapping, allowing wildcard subdomains (`*.zayjar.com`) and validated custom enterprise domains (e.g., `ordering.gourmetburger.com`) to resolve to the same underlying load balancers.
- **Edge Assets (CloudFront CDN):** Caches and delivers static website builds, optimized product images, and UI styling sheets close to users worldwide.
- **Load Balancers (AWS ALB):** Decrypts incoming HTTPS traffic, manages SSL certificates, monitors container health, and distributes traffic evenly across container clusters.
- **Container Runtime (ECS Fargate):** Runs backend APIs and queue processors in serverless, isolated containers. Containers scale out automatically based on active CPU and memory usage.
- **Database Engine (AWS RDS PostgreSQL):** Deployed as a Multi-AZ database cluster, maintaining a primary write instance in one availability zone and synchronous replica instances in other availability zones for high availability and automatic failover.
- **Connection Pools (PgBouncer):** Placed in front of PostgreSQL to manage connection limits and reuse active database connections efficiently during traffic spikes.
- **Cache Cluster (Redis Enterprise):** Configured for high availability to handle system-wide query caching, background queue management, and real-time websocket synchronization.

## 1.2 Component Architecture
The Zayjar SaaS Platform is split into logical components, enabling development teams to build, deploy, and scale system modules without affecting other parts of the application:

1. **User Client Interfaces (Frontends):**
   - **Super Admin Panel:** A secure React portal used by platform administrators to monitor tenant health, review system metrics, manage subscription billing, and audit operations.
   - **Tenant Backoffice Portal:** A Next.js web application for restaurant managers to design menus, configure branch details, manage employees, and access sales reports.
   - **Progressive Cashier Web App (PWA):** A touch-friendly cashier terminal web application optimized for fast counter orders, split-billing, and local receipt printing.
   - **Kitchen Display System (KDS):** A real-time web application for kitchen screens that displays active cooking tickets and allows kitchen staff to manage order states.
   - **Customer QR Ordering Portal:** A fast, server-side rendered (SSR) Next.js web application optimized for mobile web browsers, allowing customers to scan a table's QR code, browse the menu, and complete checkouts.

2. **Core API Gateway:**
   - Coordinates all client requests. The gateway handles JWT credential verification, checks IP-based rate limits, manages CORS headers, and routes validated requests to the core application containers.

3. **Core Backend Application (NestJS):**
   - A modular application grouping business domains (including Tenant Management, Menu Engine, Order Processing, and Kitchen Syncing) into self-contained modules.

4. **Background Worker Pool (BullMQ):**
   - An independent container group that processes long-running background tasks (including image resizing, transactional emails, SMS alerts, and payment webhooks) asynchronously.

## 1.3 Frontend Architecture
The client interfaces are written in **TypeScript** using **Next.js 14+ (App Router)** as the foundational framework, selecting the best rendering strategy for each portal's performance requirements:

- **Customer QR Menu (Server-Side Rendered - SSR):** When a customer scans a table's QR code, Next.js fetches the active menu, branch details, and table configuration from the database and compiles the page on the server. This guarantees sub-second initial page loads on mobile connections and optimizes SEO ranking for public-facing search engines.
- **Backoffice & Admin Panels (Client-Side Rendered - CSR):** Administrative dashboards load static templates securely behind authentication guards. Caching and server state synchronization are managed using **TanStack Query (React Query)**, configured with a 5-minute stale-time limit and background pre-fetching policies to ensure instant UI transitions.
- **Cashier Terminal PWA (Offline-First):** The cashier terminal uses service workers to cache core scripts and styles locally. ephemerally storing active transactions in **IndexedDB** to support offline operations. If connection drops, the cashier can continue logging sales; when connection is restored, the service worker syncs the local records with the cloud database.
- **Tailwind CSS & Radix UI UI Components:** Styling is managed globally with utility classes in **Tailwind CSS**. UI components (including modals, tooltips, and forms) are built on top of **Radix UI** primitives to ensure strict accessibility (WAI-ARIA) and keyboard navigation compliance.

## 1.4 Backend Architecture
The backend application layer is written in **NestJS**, utilizing its dependency injection container, modular architecture patterns, and structured execution pipeline:

- **Controller Layer:** Coordinates incoming HTTP and WebSocket requests. Controllers use `class-validator` pipes to validate incoming JSON structures and sanitize request bodies.
- **Service Domain Layer:** Implements business rules and transactional logic, coordinating data mutations and database transaction scopes.
- **Data Persistence (Prisma ORM):** Connects to PostgreSQL databases. Rather than standard unconstrained queries, Prisma requests are wrapped in middleware that injects tenant isolation filters dynamically.
- **Global Pipes and Interceptors:**
  - **Dynamic Validation Pipe:** Rejects payloads containing fields not explicitly defined in request DTOs, preventing parameter injection attacks.
  - **Execution Logging Interceptor:** Appends a unique correlation ID (`X-Correlation-ID`) to every request's logs, tracking execution times and trace paths.
  - **Database Exception Filter:** Captures database errors and maps them to descriptive HTTP status codes, hiding raw database error details in production.

## 1.5 Database Architecture
Persistent storage is handled by **PostgreSQL (v15+)**, leveraging its relational robustness, concurrent indexing capabilities, and structured JSONB data types:

- **Primary & Replica Routing:** The backend initializes separate write and read database connections:
  - Write queries are sent directly to the PgBouncer connection pool of the primary RDS instance.
  - Read-only queries (such as customer menu views) are sent to the PgBouncer connection pools of the RDS read replicas, reducing resource usage on the primary database.
- **Historical Partitioning:** To maintain quick search indexing and reduce search latencies, the `orders` and `order_items` tables are logically partitioned by year based on the `created_at` timestamp.
- **JSONB Customization Storage:** Dynamic UI branding options, translations, and layout configurations are stored in JSONB columns, allowing restaurants to customize their setups without requiring schema migrations.

## 1.6 Authentication Architecture
Authentication is stateless and uses **JSON Web Tokens (JWT)**:

- **Tokens Division:**
  - **Access Token:** Holds basic user identity fields, active roles, and permissions. Access tokens are signed using the asymmetric **RS256** algorithm, enabling services to verify tokens locally with the public key. Access tokens have an expiration limit of exactly 15 minutes.
  - **Refresh Token:** A secure, high-entropy random string with a 7-day expiration limit. Refresh tokens are hashed and stored in the database, and sent to client browsers inside an encrypted, HttpOnly cookie with strict security flags: `HostOnly; Secure; SameSite=Strict; Path=/; __Host-Prefix`.
- **Token Rotation & Revocation Security:**
  - When the client's Access Token expires, it issues an HTTP request to `/api/v1/auth/refresh`, delivering the cookie-encapsulated Refresh Token.
  - The authentication system performs **Refresh Token Rotation (RTR)**. Every refresh cycle generates a new refresh token and invalidates the previous one. If a token is reused, the system flags it as potential theft, invalidates the entire token tree for that user, blacklists all active sessions in Redis, and forces a manual login.

## 1.7 Authorization Architecture
Authorization is executed using a hybridized **Role-Based Access Control (RBAC)** and **Attribute-Based Access Control (ABAC)** model:

- **Role Levels:**
  - `PLATFORM_OWNER`: Has complete administrative permissions across the SaaS platform.
  - `RESTAURANT_OWNER`: Has full administrative control over their tenant's configuration.
  - `BRANCH_MANAGER`: Managed permissions restricted to their assigned branch location.
  - `CASHIER`: Scoped to order registration, split billing, and cashier terminal management.
  - `KITCHEN_STAFF`: Scoped to reading kitchen tickets and updating cooking statuses on KDS screens.
  - `CUSTOMER`: Scoped to reading menus and managing their shopping cart and checkout processes.
- **Dynamic Policy Engine:**
  - API endpoints are protected using NestJS Guards and the **CASL** library, which evaluates ABAC permissions dynamically (e.g., verifying that a branch manager can only modify products linked to their assigned branch).

## 1.8 Multi-Tenant Architecture
The platform is designed around a **Single Shared Database with Logical Tenant Isolation**:

- **Explicit Database Scoping:** Every tenant-scoped table contains a mandatory, non-nullable `tenant_id` column.
- **Dynamic Context Injection:** Incoming requests pass through middleware that resolves the active tenant ID from the subdomain (e.g., `burgers.zayjar.com`) or custom domain (e.g., `menu.burgers.com`).
- **Query Filter Enforcement:** The Prisma ORM is wrapped in middleware that automatically appends a `tenant_id` filter to all database queries, ensuring users cannot view or modify other tenants' data.

## 1.9 Restaurant Isolation
To enforce strict boundary isolation between restaurants and prevent cross-tenant security events, the following controls are deployed:

- **JSON Web Token Scoping:** A user's JWT payload is cryptographically bound to a single tenant ID. If a user tries to invoke an operation on a restaurant with a payload that references a different tenant, the API gateway rejects the token before it executes on the core application server.
- **Query Interceptor Isolation:** All read, update, delete, and insert operations are wrapped in an application interceptor. The query parser parses runtime queries, validating that all target mutations explicitly enforce the authenticated user's assigned `tenant_id` context. If a query lacks a `tenant_id` filter or targets a mismatched ID, the interceptor blocks the query and flags a security violation.
- **Cross-Tenant Prevention Testing:** The automated test runner runs test sequences that attempt cross-tenant resource reads and mutations (e.g., Tenant B attempting to read Tenant A's private orders). These security tests must pass on every CI build to verify data isolation.

## 1.10 Subscription Architecture
The SaaS platform uses **Stripe Billing** to manage subscription plans and monetize services:

- **Subscription Status States:** A tenant's subscription status is tracked in real-time through an active state machine:
  - `TRIALING`: Fully featured, free-access sandbox environment, expiring after 14 days.
  - `ACTIVE`: Normal operating state, payment processed successfully.
  - `PAST_DUE`: Automated billing failed. The system initiates a 7-day grace period, sending email notifications, but does not disable tenant features.
  - `UNPAID`: Payment has repeatedly failed. Access to backoffice panels is restricted, and guest ordering is paused.
  - `CANCELED`: Core access is disabled, resources are frozen for 30 days before clean-up, and all subdomains are released.
- **Feature Gating Middleware:**
  - Standard guards evaluate active plan parameters on every request. If a Silver-tier tenant attempts to add a fourth branch or register a custom domain (a Gold-tier feature), the backend blocks the operation with an HTTP 403 Forbidden response.

# 8. THIRD-PARTY INTEGRATIONS & STACK COLLATERAL

## 8.1 GitHub & Monorepository Configuration
The project is organized in a single monorepository hosted on GitHub, enabling teams to coordinate changes and manage core codebases, tools, and infrastructure in one place:
- **Repository Organization:** The workspace uses a **pnpm workspaces** topology, grouping projects into separate directories:
  - `apps/api`: Core NestJS backend services.
  - `apps/backoffice`: Next.js backoffice and admin portal client.
  - `apps/qr-menu`: Next.js customer ordering and QR menu web application.
  - `packages/db`: Shared Prisma schemas, migrations, and database clients.
  - `packages/types`: Shared TypeScript definitions and DTO definitions.
- **GitHub Actions Pipelines:** Branch protection rules require all commits to pass pull request checks (including ESLint checks, automated unit test suites, and safety analysis scans) before they can be merged into the `main` development branch.

---

## 8.2 Stripe Core Payments & Billing Engine
Subscription billing and customer checkouts are integrated directly with **Stripe**:
- **Stripe Billing Integration:** The SaaS platform uses **Stripe Billing** to manage subscriptions. New tenants are mapped to Stripe customer records, and their subscriptions are tied to Stripe price IDs.
- **Payment Intents API:** When an order is placed, the payment engine uses the Stripe Payment Intents API to securely process and verify transactions.
- **Webhook Handlers:** To keep subscription statuses up-to-date, a webhook controller listens for Stripe event dispatches. When events (such as `invoice.payment_succeeded`, `invoice.payment_failed`, or `customer.subscription.deleted`) are received, the webhook service updates the matching tenant's status and access permissions in the database instantly.

## 8.3 Regional Wallet Integrations (Apple Pay, Google Pay, Local Wallets)
To optimize checkout experiences across different markets, the payment engine supports regional wallet integrations:
- **Mobile Wallets Support:** The Stripe Checkout client is configured to support mobile payment methods (including **Apple Pay** and **Google Pay**).
- **Middle Eastern Payment Integrations:** For Middle Eastern markets, the payment gateway integrates with local payment systems (such as **KNET** in Kuwait, **Benefit** in Bahrain, and **Mada** in Saudi Arabia) through regional integrations (including **Tap Payments** and **PayTabs**), ensuring high transactional authorization rates.

---

## 8.4 Telemetry, Logging (Winston, ELK Stack), & APM (Datadog)
The platform uses centralized logging, metrics tracking, and APM tools to maintain system visibility and monitor health:
- **Centralized Application Logging:** The NestJS API uses the **Winston** library to output structured logs in JSON format. Logs are enriched with request correlation IDs and sent to an **ELK Stack** (Elasticsearch, Logstash, Kibana) for centralized indexing and analysis.
- **Application Performance Monitoring (APM):** **Datadog APM** is integrated into container runtimes. APM agents track system performance metrics (including HTTP route latency, SQL query execution durations, external API timeouts, and memory utilization) to help identify and debug performance bottlenecks.

## 8.5 Automated Backups & Disaster Recovery Strategy
To prevent data loss and ensure system recovery, the database applies automated backup and replication policies:
- **Automated Backup Policies:**
  - **AWS RDS Multi-AZ Replication:** Relational databases are deployed in a Multi-AZ configuration, maintaining a synchronous standby mirror in a separate availability zone for immediate failover.
  - **Automated Daily Snapshots:** The database executes automated daily snapshots, retaining recovery copies for exactly 35 days.
  - **Write-Ahead Log (WAL) Archiving:** PostgreSQL WAL logs are continuously streamed and archived to S3 storage, enabling Point-in-Time Recovery (PITR) to any second within the retention window.
- **Disaster Recovery Targets:**
  - Recovery Time Objective (RTO): Less than 15 minutes (ensuring services are restored quickly).
  - Recovery Point Objective (RPO): Less than 5 minutes (minimizing potential data loss).

# 3. REST API SPECIFICATION

## 3.1 Global Conventions & Request Headers
The Zayjar REST API follows strict RESTful conventions using HTTP methods (`GET`, `POST`, `PUT`, `DELETE`) to denote operations. Payload data is exchanged exclusively in JSON format.

### Standard Request Headers
- `Content-Type`: Must be set to `application/json` for all mutating actions.
- `Authorization`: `Bearer <JWT_ACCESS_TOKEN>` - Mandated for all secure operations.
- `X-Tenant-ID`: UUIDv4 string specifying target tenant scope (used to resolve context on multi-tenant operations).
- `X-Branch-ID`: UUIDv4 string specifying target branch scope.
- `X-Correlation-ID`: Unique request ID (e.g., `req-9a1b-cd34`) to trace logs across distributed components.

### Standard Error Responses (HTTP 4xx / 500)
Every API error response follows this structured JSON pattern:
```json
{
  "statusCode": 400,
  "timestamp": "2026-07-20T17:52:00.000Z",
  "path": "/api/v1/menu/products",
  "error": "Bad Request",
  "message": [
    "base_price must be a positive number",
    "category_id must be a valid UUID"
  ],
  "correlationId": "req-9a1b-cd34"
}
```

## 3.2 Authentication & Session Endpoints

### 3.2.1 POST /api/v1/auth/login
- **Purpose**: Authenticates administrative portal users, managers, and cashiers, returning an access token and setting a secure HTTP-Only refresh cookie.
- **Authentication**: None.
- **Rate Limit**: 10 attempts per minute per IP.
- **Request Body**:
  ```json
  {
    "email": "owner@restaurantbrand.com",
    "password": "SecurePassword123!",
    "mfaToken": "123456"
  }
  ```
  - `email` (string, required): Lowercase, valid format.
  - `password` (string, required): Min 8, max 64 characters.
  - `mfaToken` (string, optional): 6-digit TOTP token if MFA is enabled.
- **Success Response (HTTP 200 OK)**:
  ```json
  {
    "accessToken": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
    "expiresIn": 900,
    "user": {
      "id": "u410c-9a1b-42b8-bf83-097a18fcd34c",
      "tenantId": "7a18f-39b0-4050-bf83-097a18fcd34b",
      "firstName": "John",
      "lastName": "Doe",
      "email": "owner@restaurantbrand.com",
      "roles": ["RESTAURANT_OWNER"],
      "mfaRequired": false
    }
  }
  ```
  *Cookie set:* `__Host-Refresh-Token=<cryptographic_token>; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=604800`
- **Error Responses**:
  - `HTTP 400 Bad Request`: Payload validation failure.
  - `HTTP 401 Unauthorized`: Invalid credentials or incorrect MFA token.
  - `HTTP 429 Too Many Requests`: Rate limit exceeded.

### 3.2.2 POST /api/v1/auth/refresh
- **Purpose**: Rotates the user's active session, returning a new access token and a refreshed HTTP-Only cookie.
- **Authentication**: Uses the `__Host-Refresh-Token` cookie.
- **Rate Limit**: 60 requests per minute per IP.
- **Success Response (HTTP 200 OK)**:
  ```json
  {
    "accessToken": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
    "expiresIn": 900
  }
  ```
  *Cookie set:* `__Host-Refresh-Token=<new_cryptographic_token>; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=604800`
- **Error Responses**:
  - `HTTP 401 Unauthorized`: Mismatched or revoked refresh token.
  - `HTTP 403 Forbidden`: Token reuse detected (triggers tree invalidation).

### 3.2.3 POST /api/v1/auth/logout
- **Purpose**: Terminated the user's active session, revokes the refresh token in Redis, and clears the client's cookie.
- **Authentication**: Bearer Access Token & Refresh Cookie.
- **Success Response (HTTP 200 OK)**:
  ```json
  {
    "success": true,
    "message": "Session terminated successfully."
  }
  ```
  *Cookie cleared:* `__Host-Refresh-Token=; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=0`

### 3.2.4 GET /api/v1/auth/me
- **Purpose**: Returns the authenticated user's profile and active permissions mapping.
- **Authentication**: Bearer Access Token.
- **Success Response (HTTP 200 OK)**:
  ```json
  {
    "user": {
      "id": "u410c-9a1b-42b8-bf83-097a18fcd34c",
      "tenantId": "7a18f-39b0-4050-bf83-097a18fcd34b",
      "firstName": "John",
      "lastName": "Doe",
      "email": "owner@restaurantbrand.com",
      "roles": ["RESTAURANT_OWNER"],
      "permissions": [
        "restaurant:read",
        "restaurant:update",
        "menu:create",
        "menu:read",
        "menu:update",
        "menu:delete",
        "orders:read"
      ]
    }
  }
  ```

### 3.2.5 POST /api/v1/auth/mfa/enable
- **Purpose**: Generates a TOTP secret and QR code for two-factor authentication setup.
- **Authentication**: Bearer Access Token.
- **Success Response (HTTP 200 OK)**:
  ```json
  {
    "secret": "NBSWY3DPEB3W64TBNQ",
    "qrCodeDataUrl": "data:image/png;base64,iVBORw0KGgoAAAANS..."
  }
  ```

### 3.2.6 POST /api/v1/auth/mfa/verify
- **Purpose**: Verifies and activates MFA on the user's account using the generated secret.
- **Authentication**: Bearer Access Token.
- **Request Body**:
  ```json
  {
    "mfaToken": "123456"
  }
  ```
- **Success Response (HTTP 200 OK)**:
  ```json
  {
    "mfaEnabled": true,
    "backupCodes": [
      "code1-abcd-efgh",
      "code2-ijkl-mnop",
      "code3-qrst-uvwx"
    ]
  }
  ```

```
                  +----------------------------------------+
                                       |
                +----------------------+----------------------+
                |                                             |
                v                                             v
  [ Database Constraints Fail ]                [ Application Logic Reject ]
  - Unique Violations -> HTTP 409              - Insufficient Funds -> HTTP 402
  - Missing Target -> HTTP 404                 - Out of Stock -> HTTP 422
  - Check Failures -> HTTP 400                 - Security Breach -> HTTP 403
```

- **Database-to-HTTP Status Code Mapping Matrix:**
  - `P2002` (Prisma Unique Constraint Violation): Maps to `HTTP 409 Conflict`.
  - `P2003` (Prisma Foreign Key Constraint Fail): Maps to `HTTP 404 Not Found` (referenced entity does not exist) or `HTTP 400 Bad Request` (referenced delete restricted).
  - `P2025` (Prisma Record Not Found): Maps to `HTTP 404 Not Found`.
- **Validation Pipe Filters:** If the JSON body lacks a required property or fails validation checks, the NestJS pipeline returns an `HTTP 400 Bad Request` response with an array detailing every validation error.

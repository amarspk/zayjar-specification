# 1. SYSTEM ARCHITECTURE

## 1.1 High-Level Architecture
The Zayjar Restaurant SaaS Platform is built using a modern, multi-tenant cloud-native architecture designed to ensure maximum isolation, operational resilience, and sub-second execution speeds. The system uses a logical domain separation model (modular monolith) implemented in **NestJS (Node.js)** for the backend layer, coupled with high-availability relational databases and containerized background worker clusters.

### Infrastructure Core Services:
- **DNS & Edge Routing (Route 53):** Handles dynamic tenant mapping, allowing wildcard subdomains (`*.zayjar.com`) and validated custom enterprise domains (e.g., `ordering.gourmetburger.com`) to resolve to the same underlying load balancers.
- **Edge Assets (CloudFront CDN):** Caches and delivers static website builds, optimized product images, and UI styling sheets close to users worldwide.
- **Load Balancers (AWS ALB):** Decrypts incoming HTTPS traffic, manages SSL certificates, monitors container health, and distributes traffic evenly across container clusters.
- **Container Runtime (ECS Fargate):** Runs backend APIs and queue processors in serverless, isolated containers. Containers scale out automatically based on active CPU and memory usage.
- **Database Engine (AWS RDS PostgreSQL):** Deployed as a Multi-AZ database cluster, maintaining a primary write instance in one availability zone and synchronous replica instances in other availability zones for high availability and automatic failover.
- **Connection Pools (PgBouncer):** Placed in front of PostgreSQL to manage connection limits and reuse active database connections efficiently during traffic spikes.
- **Cache Cluster (Redis Enterprise):** Configured for high availability to handle system-wide query caching, background queue management, and real-time websocket synchronization.

## 1.2 Component Architecture
The Zayjar SaaS Platform is split into logical components, enabling development teams to build, deploy, and scale system modules without affecting other parts of the application:

1. **User Client Interfaces (Frontends):**
   - **Super Admin Panel:** A secure React portal used by platform administrators to monitor tenant health, review system metrics, manage subscription billing, and audit operations.
   - **Tenant Backoffice Portal:** A Next.js web application for restaurant managers to design menus, configure branch details, manage employees, and access sales reports.
   - **Progressive Cashier Web App (PWA):** A touch-friendly cashier terminal web application optimized for fast counter orders, split-billing, and local receipt printing.
   - **Kitchen Display System (KDS):** A real-time web application for kitchen screens that displays active cooking tickets and allows kitchen staff to manage order states.
   - **Customer QR Ordering Portal:** A fast, server-side rendered (SSR) Next.js web application optimized for mobile web browsers, allowing customers to scan a table's QR code, browse the menu, and complete checkouts.

2. **Core API Gateway:**
   - Coordinates all client requests. The gateway handles JWT credential verification, checks IP-based rate limits, manages CORS headers, and routes validated requests to the core application containers.

3. **Core Backend Application (NestJS):**
   - A modular application grouping business domains (including Tenant Management, Menu Engine, Order Processing, and Kitchen Syncing) into self-contained modules.

4. **Background Worker Pool (BullMQ):**
   - An independent container group that processes long-running background tasks (including image resizing, transactional emails, SMS alerts, and payment webhooks) asynchronously.

## 1.3 Frontend Architecture
The client interfaces are written in **TypeScript** using **Next.js 14+ (App Router)** as the foundational framework, selecting the best rendering strategy for each portal's performance requirements:

- **Customer QR Menu (Server-Side Rendered - SSR):** When a customer scans a table's QR code, Next.js fetches the active menu, branch details, and table configuration from the database and compiles the page on the server. This guarantees sub-second initial page loads on mobile connections and optimizes SEO ranking for public-facing search engines.
- **Backoffice & Admin Panels (Client-Side Rendered - CSR):** Administrative dashboards load static templates securely behind authentication guards. Caching and server state synchronization are managed using **TanStack Query (React Query)**, configured with a 5-minute stale-time limit and background pre-fetching policies to ensure instant UI transitions.
- **Cashier Terminal PWA (Offline-First):** The cashier terminal uses service workers to cache core scripts and styles locally. ephemerally storing active transactions in **IndexedDB** to support offline operations. If connection drops, the cashier can continue logging sales; when connection is restored, the service worker syncs the local records with the cloud database.
- **Tailwind CSS & Radix UI UI Components:** Styling is managed globally with utility classes in **Tailwind CSS**. UI components (including modals, tooltips, and forms) are built on top of **Radix UI** primitives to ensure strict accessibility (WAI-ARIA) and keyboard navigation compliance.

## 1.4 Backend Architecture
The backend application layer is written in **NestJS**, utilizing its dependency injection container, modular architecture patterns, and structured execution pipeline:

- **Controller Layer:** Coordinates incoming HTTP and WebSocket requests. Controllers use `class-validator` pipes to validate incoming JSON structures and sanitize request bodies.
- **Service Domain Layer:** Implements business rules and transactional logic, coordinating data mutations and database transaction scopes.
- **Data Persistence (Prisma ORM):** Connects to PostgreSQL databases. Rather than standard unconstrained queries, Prisma requests are wrapped in middleware that injects tenant isolation filters dynamically.
- **Global Pipes and Interceptors:**
  - **Dynamic Validation Pipe:** Rejects payloads containing fields not explicitly defined in request DTOs, preventing parameter injection attacks.
  - **Execution Logging Interceptor:** Appends a unique correlation ID (`X-Correlation-ID`) to every request's logs, tracking execution times and trace paths.
  - **Database Exception Filter:** Captures database errors and maps them to descriptive HTTP status codes, hiding raw database error details in production.

## 1.5 Database Architecture
Persistent storage is handled by **PostgreSQL (v15+)**, leveraging its relational robustness, concurrent indexing capabilities, and structured JSONB data types:

- **Primary & Replica Routing:** The backend initializes separate write and read database connections:
  - Write queries are sent directly to the PgBouncer connection pool of the primary RDS instance.
  - Read-only queries (such as customer menu views) are sent to the PgBouncer connection pools of the RDS read replicas, reducing resource usage on the primary database.
- **Historical Partitioning:** To maintain quick search indexing and reduce search latencies, the `orders` and `order_items` tables are logically partitioned by year based on the `created_at` timestamp.
- **JSONB Customization Storage:** Dynamic UI branding options, translations, and layout configurations are stored in JSONB columns, allowing restaurants to customize their setups without requiring schema migrations.

## 1.6 Authentication Architecture
Authentication is stateless and uses **JSON Web Tokens (JWT)**:

- **Tokens Division:**
  - **Access Token:** Holds basic user identity fields, active roles, and permissions. Access tokens are signed using the asymmetric **RS256** algorithm, enabling services to verify tokens locally with the public key. Access tokens have an expiration limit of exactly 15 minutes.
  - **Refresh Token:** A secure, high-entropy random string with a 7-day expiration limit. Refresh tokens are hashed and stored in the database, and sent to client browsers inside an encrypted, HttpOnly cookie with strict security flags: `HostOnly; Secure; SameSite=Strict; Path=/; __Host-Prefix`.
- **Token Rotation & Revocation Security:**
  - When the client's Access Token expires, it issues an HTTP request to `/api/v1/auth/refresh`, delivering the cookie-encapsulated Refresh Token.
  - The authentication system performs **Refresh Token Rotation (RTR)**. Every refresh cycle generates a new refresh token and invalidates the previous one. If a token is reused, the system flags it as potential theft, invalidates the entire token tree for that user, blacklists all active sessions in Redis, and forces a manual login.

## 1.7 Authorization Architecture
Authorization is executed using a hybridized **Role-Based Access Control (RBAC)** and **Attribute-Based Access Control (ABAC)** model:

- **Role Levels:**
  - `PLATFORM_OWNER`: Has complete administrative permissions across the SaaS platform.
  - `RESTAURANT_OWNER`: Has full administrative control over their tenant's configuration.
  - `BRANCH_MANAGER`: Managed permissions restricted to their assigned branch location.
  - `CASHIER`: Scoped to order registration, split billing, and cashier terminal management.
  - `KITCHEN_STAFF`: Scoped to reading kitchen tickets and updating cooking statuses on KDS screens.
  - `CUSTOMER`: Scoped to reading menus and managing their shopping cart and checkout processes.
- **Dynamic Policy Engine:**
  - API endpoints are protected using NestJS Guards and the **CASL** library, which evaluates ABAC permissions dynamically (e.g., verifying that a branch manager can only modify products linked to their assigned branch).

## 1.8 Multi-Tenant Architecture
The platform is designed around a **Single Shared Database with Logical Tenant Isolation**:

- **Explicit Database Scoping:** Every tenant-scoped table contains a mandatory, non-nullable `tenant_id` column.
- **Dynamic Context Injection:** Incoming requests pass through middleware that resolves the active tenant ID from the subdomain (e.g., `burgers.zayjar.com`) or custom domain (e.g., `menu.burgers.com`).
- **Query Filter Enforcement:** The Prisma ORM is wrapped in middleware that automatically appends a `tenant_id` filter to all database queries, ensuring users cannot view or modify other tenants' data.

## 1.9 Restaurant Isolation
To enforce strict boundary isolation between restaurants and prevent cross-tenant security events, the following controls are deployed:

- **JSON Web Token Scoping:** A user's JWT payload is cryptographically bound to a single tenant ID. If a user tries to invoke an operation on a restaurant with a payload that references a different tenant, the API gateway rejects the token before it executes on the core application server.
- **Query Interceptor Isolation:** All read, update, delete, and insert operations are wrapped in an application interceptor. The query parser parses runtime queries, validating that all target mutations explicitly enforce the authenticated user's assigned `tenant_id` context. If a query lacks a `tenant_id` filter or targets a mismatched ID, the interceptor blocks the query and flags a security violation.
- **Cross-Tenant Prevention Testing:** The automated test runner runs test sequences that attempt cross-tenant resource reads and mutations (e.g., Tenant B attempting to read Tenant A's private orders). These security tests must pass on every CI build to verify data isolation.

## 1.10 Subscription Architecture
The SaaS platform uses **Stripe Billing** to manage subscription plans and monetize services:

- **Subscription Status States:** A tenant's subscription status is tracked in real-time through an active state machine:
  - `TRIALING`: Fully featured, free-access sandbox environment, expiring after 14 days.
  - `ACTIVE`: Normal operating state, payment processed successfully.
  - `PAST_DUE`: Automated billing failed. The system initiates a 7-day grace period, sending email notifications, but does not disable tenant features.
  - `UNPAID`: Payment has repeatedly failed. Access to backoffice panels is restricted, and guest ordering is paused.
  - `CANCELED`: Core access is disabled, resources are frozen for 30 days before clean-up, and all subdomains are released.
- **Feature Gating Middleware:**
  - Standard guards evaluate active plan parameters on every request. If a Silver-tier tenant attempts to add a fourth branch or register a custom domain (a Gold-tier feature), the backend blocks the operation with an HTTP 403 Forbidden response.

# 2. DATABASE SCHEMA SPECIFICATION

## 2.1 Schema Overview & Design Philosophy
The relational schema is built on **PostgreSQL 15+** to maintain strict data integrity, fast query speeds, and zero-downtime migrations. 
- All table primary keys are standard **UUIDv4** strings to prevent ID enumeration attacks and simplify distributed data ingestion.
- Foreign keys explicitly enforce relational integrity constraints with structured delete rules (such as cascade or restrict actions).
- Every record features tracking columns for debugging and auditing purposes: `created_at` (timestamp with timezone), `updated_at` (timestamp with timezone), and `deleted_at` (for soft deletes).

## 2.2 Table-by-Table Data Dictionary

### 2.2.1 Table: `tenants`
Stores the high-level customer organizations subscribing to the SaaS platform.
- **Columns:**
  - `id` | `UUID` | `NOT NULL` | `PRIMARY KEY` | Default: `uuid_generate_v4()` | Unique tenant identifier.
  - `name` | `VARCHAR(255)` | `NOT NULL` | The registered corporate name of the tenant organization.
  - `subdomain` | `VARCHAR(63)` | `NOT NULL` | Unique subdomain registered on the platform (e.g., `burgers`).
  - `custom_domain` | `VARCHAR(255)` | `NULL` | Optional verified custom domain mapped by the tenant.
  - `status` | `tenant_status` (ENUM) | `NOT NULL` | Default: `'TRIALING'` | Active operations status.
  - `logo_url` | `VARCHAR(2048)` | `NULL` | Cloudfront CDN URL pointing to the tenant's brand logo image.
  - `banner_url` | `VARCHAR(2048)` | `NULL` | Cloudfront CDN URL pointing to the tenant's banner cover image.
  - `primary_color` | `VARCHAR(7)` | `NOT NULL` | Default: `'#000000'` | Brand primary hex color.
  - `secondary_color` | `VARCHAR(7)` | `NOT NULL` | Default: `'#FFFFFF'` | Brand secondary hex color.
  - `stripe_customer_id` | `VARCHAR(255)` | `NULL` | Stripe billing integration pointer.
  - `created_at` | `TIMESTAMPTZ` | `NOT NULL` | Default: `NOW()` | Creation timestamp.
  - `updated_at` | `TIMESTAMPTZ` | `NOT NULL` | Default: `NOW()` | Modification timestamp.
  - `deleted_at` | `TIMESTAMPTZ` | `NULL` | Soft delete timestamp.
- **Indexes:**
  - `idx_tenants_subdomain` (B-Tree, UNIQUE) on `subdomain` (WHERE `deleted_at IS NULL`).
  - `idx_tenants_custom_domain` (B-Tree, UNIQUE) on `custom_domain` (WHERE `custom_domain IS NOT NULL` AND `deleted_at IS NULL`).
  - `idx_tenants_stripe_customer` (B-Tree) on `stripe_customer_id`.
- **Foreign Keys:**
  - None.
- **Constraints:**
  - `chk_tenants_subdomain_format`: `subdomain ~* '^[a-z0-9-]+$'` (Allows alphanumeric characters and hyphens only).
  - `chk_tenants_primary_color`: `primary_color ~* '^#[0-9a-fA-F]{6}$'` (Strict Hex validation).
  - `chk_tenants_secondary_color`: `secondary_color ~* '^#[0-9a-fA-F]{6}$'` (Strict Hex validation).
- **Triggers:**
  - `trg_tenants_set_updated_at`: Executes on `BEFORE UPDATE` to reset `updated_at` to `NOW()`.

## 2.3 Global Relationships & ERD Specifications
The relational logic of the database is structured around high-integrity relationships designed to prevent dangling pointers:
1. **Tenant Isolation Core:** Almost all transactional tables cascade back to the `tenants` table. Dropping a tenant executes a system-wide logical cascade that cleans up all linked tables.
2. **Menu Item Pricing Trees:** `products` have many `product_sizes`, `product_variants`, and `product_addons`. The pricing configuration forms an inheritance structure: base product prices are overridden by sizes and variants.
3. **Checkout Order Lines:** `orders` contain multiple `order_items`, which in turn hold `order_item_addons`. To protect historical sales statistics, products and variants linked to order items are configured with `ON DELETE RESTRICT` constraints. Menu deletions are soft-deletes; the original relational objects remain in the DB to avoid breaking billing reports.

## 2.4 Indexing Strategy & Performance Profiling
Indices are optimized to maintain system performance as the platform scales:
- **Foreign Keys:** B-Tree indexes are explicitly defined on all foreign keys to prevent sequential scans during join queries.
- **Unique Scopes:** Tenant uniqueness bounds are enforced using partial indexes that ignore soft-deleted records (e.g., `WHERE deleted_at IS NULL`).
- **Full-Text Menu Searching:** A PostgreSQL GIN index is defined on a generated `tsv_menu_search` column inside the `products` table (combining name, description, and category name attributes) to enable instant menu search queries on client apps.
- **Order Tracking:** Composite indexes are defined on (`tenant_id`, `branch_id`, `status`, `created_at`) to optimize KDS polling operations.

## 2.6 Constraints (Check, Unique, Complex Exclusions)
Constraints prevent invalid database states:
- **Financial Fields:** Price checks require financial columns to be greater than or equal to 0 (e.g., `CHECK (base_price >= 0.00)` on `products`).
- **Latitude & Longitude Boundaries:** Spatial checks on coordinates (e.g., `CHECK (latitude BETWEEN -90 AND 90)`).
- **Subdomain Routing Standards:** Format rules prevent subdomains from including special symbols, periods, or capitals.
- **Custom Domain Rules:** Custom domains must resolve with standard host format validations.
- **Add-on Limits Check:** Custom validator checks ensure addon groups conform to selection rules: `CHECK (max_selections >= min_selections)`.

## 2.7 Database Triggers & Lifecycle Hooks
Triggers are defined to handle timestamp updates and write operations automatically:
- **Auto Update Timestamps:** Every table containing an `updated_at` column runs an automated trigger that sets `updated_at = NOW()` whenever any column is modified.
- **Structured Audit Logs Integration:** The database records critical operations by writing change details (such as previous and updated states) into the `audit_logs` table via triggers on `UPDATE` or `DELETE` statements on tables like `products` or `orders`.
- **System Notification Dispatchers:** Order status modifications trigger asynchronous notifications. When an order status updates to `'READY'`, a trigger inserts a record into the notifications queue, which is then broadcast to the customer-facing frontend.

## 2.8 Deletion, Archiving, and Soft-Delete Policies
To protect transactional history, data removal processes are strictly controlled:
- **Soft Delete Policy:** Tables representing operational data (including `tenants`, `users`, `restaurants`, `branches`, `categories`, `products`, `customers`, and `tables`) use a logical soft-delete mechanism. These tables feature a nullable `deleted_at` timestamp. Queries to these tables must include a `deleted_at IS NULL` filter to exclude soft-deleted records.
- **Hard Delete Scopes:** Junction tables (such as `user_roles` and `role_permissions`) use hard deletes. Removing a role deletes its mapped association records.
- **Relational Restriction Rules:** Database restrictions prevent the deletion of core menu components that are linked to historical orders. If a restaurant manager tries to delete a product linked to an existing order, the operation is blocked. Instead, the UI prompts the user to archive or soft-delete the product, which hides it from active client menus while preserving historical order records.
- **Partition Archiving Policies:** At the end of each fiscal year, historical order partition tables are detached, compressed into parquet files, and archived to AWS S3 Glacier Deep Archive for long-term storage and compliance.

## 2.9 Tenant Isolation & Ownership Rules
Multi-tenant security isolation rules are enforced at the database level:
- **Explicit Ownership Scopes:** Every operational record is linked to a tenant via a mandatory, non-nullable `tenant_id` foreign key.
- **Database Row-Level Security (RLS) Implementation:** PostgreSQL RLS is enabled on all tenant-scoped tables:
  ```sql
  ALTER TABLE products ENABLE ROW LEVEL SECURITY;
  
  CREATE POLICY tenant_isolation_policy ON products
  FOR ALL
  USING (tenant_id = current_setting('app.current_tenant_id', true)::UUID);
  ```
- **Context Injection Middleware:** When a request is processed, the backend middleware extracts the tenant ID and initiates a session-scoped variable `app.current_tenant_id` inside the database transaction context. This ensures the RLS policy automatically filters out data from other tenants, preventing unauthorized access.

The Zayjar REST API follows strict RESTful conventions using HTTP methods (`GET`, `POST`, `PUT`, `DELETE`) to denote operations. Payload data is exchanged exclusively in JSON format.

## 3.11 Standard Error Schemas & Handler Pipelines
The API applies global NestJS custom filter middleware to map server errors to clean, informative HTTP status codes.

```
                  +----------------------------------------+
                  |         EXCEPTIONAL FAILURE AREA       |
                  +----------------------------------------+
                                       |
                +----------------------+----------------------+
                |                                             |
                v                                             v
  [ Database Constraints Fail ]                [ Application Logic Reject ]
  - Unique Violations -> HTTP 409              - Insufficient Funds -> HTTP 402
  - Missing Target -> HTTP 404                 - Out of Stock -> HTTP 422
  - Check Failures -> HTTP 400                 - Security Breach -> HTTP 403
```

- **Database-to-HTTP Status Code Mapping Matrix:**
  - `P2002` (Prisma Unique Constraint Violation): Maps to `HTTP 409 Conflict`.
  - `P2003` (Prisma Foreign Key Constraint Fail): Maps to `HTTP 404 Not Found` (referenced entity does not exist) or `HTTP 400 Bad Request` (referenced delete restricted).
  - `P2025` (Prisma Record Not Found): Maps to `HTTP 404 Not Found`.
- **Validation Pipe Filters:** If the JSON body lacks a required property or fails validation checks, the NestJS pipeline returns an `HTTP 400 Bad Request` response with an array detailing every validation error.

# 4. BUSINESS LOGIC & WORKFLOWS

## 4.1 Tenant & Restaurant Onboarding Workflow
When a new restaurant owner signs up for the Zayjar Restaurant SaaS Platform, the system initiates a structured onboarding process to provision their tenant environment and prepare it for use.

```
[ Signup Requested ] 
        |
        v
[ Validate unique subdomain and email ] 
        |
        v
[ DB Transaction Block Init ] 
  ├── Insert tenant profile
  ├── Create default Subscription (TRIALING)
  ├── Register User account (RESTAURANT_OWNER)
  ├── Initialize default Restaurant Brand
  └── Create first standard Branch (Main Office)
        |
        v
[ Provisions Stripe Customer Profile ] 
        |
        v
[ Deliver Welcome Email Sequence ]
```

### Onboarding Steps:
1. **Validation Checks:** The application validates that the requested subdomain is available, conformant to naming standards, and not on the reserved subdomains list (e.g., `'admin'`, `'api'`, `'www'`). It also checks that the registered email address is unique.
2. **Database Transaction Block:** To ensure database consistency, all onboarding operations are wrapped inside a database transaction:
   - **Create Tenant:** A new `tenants` record is initialized with the company name, dynamic subdomain, and default branding configurations.
   - **Create Subscription:** A `subscriptions` record is generated, linking the tenant to the trial plan and setting a trial end date 14 days out.
   - **Register Owner Account:** A `users` record is initialized with the owner's details and cryptographically hashed password. The user is assigned the `'RESTAURANT_OWNER'` role.
   - **Create Restaurant Entity:** A `restaurants` record is generated, setting default values for taxes, currency, and timezone based on the tenant's location.
   - **Create Default Branch:** A `branches` record is generated to serve as the brand's primary branch location, pre-populated with standard operating hours templates.
3. **Stripe Profile Provisioning:** The tenant is registered in Stripe, creating a customer profile and linking its ID (`stripe_customer_id`) back to the tenant record.
4. **Welcome Sequence Dispatch:** A background worker sends a transactional welcome email sequence via SendGrid containing setup instructions and guides.

## 4.3 Menu Item Price Inheritance Engine
The menu system supports flexible pricing configurations, allowing restaurants to set base prices that can be overridden at the size or variant level.

```
                              [ Evaluate Product Price ]
                                          |
                      +-------------------+-------------------+
                      |                                       |
            [ Variant Selected? ]                     [ Size Selected? ]
                      |                                       |
             (Variant Price Base)                   (Add Base + Size Adjust)
                      |                                       |
                      +-------------------+-----------------------------------------+
                                          |                                         |
                                          v                                         v
                             [ Read Addon Item Cost ]                    [ Final Invoice Cost ]
                             - Add addon cost adjustments
```

### Inheritance Calculation Pipeline:
- **Condition A (Base Item Only):** When a product is ordered without any size modifications or variants, its price is equal to the product's `base_price`.
- **Condition B (Size Override):** When a specific size modifier is selected, the item cost is calculated as the product's base price plus the size's price adjustment:
  $$\text{Final Unit Cost} = \text{Product Base Price} + \text{Size Price Adjustment}$$
- **Condition C (Variant Override):** If the product uses inventory variants (such as a specific size/option combination), the item cost is set directly to the variant's absolute price, overriding the product's base price.
- **Condition D (Add-on Calculations):** For any selected add-on customization items, their individual costs are added to the final unit price:
  $$\text{Final Item Cost} = \text{Base calculated cost} + \sum (\text{Add-on Option prices})$$

## 4.4 Comprehensive Order Lifecycle State Machine
Orders transition through a defined set of states to ensure consistent processing and accurate tracking across the system.

```
       [ DRAFT ] 
           |
           v
      [ PENDING ]  ------( Payment Fails )-------> [ CANCELLED ]
           |
           v ( Payment Success )
      [ ACCEPTED ] 
           |
           v ( Kitchen Admits Ticket )
     [ PREPARING ] 
           |
           v ( Cooking Done )
       [ READY ] 
           |
           +-----------------------+-----------------------+
           |                       |                       |
           v ( Dine-in Serve )     v ( Takeaway Picked )   v ( Delivery Done )
     [ COMPLETED ]           [ COMPLETED ]           [ COMPLETED ]
```

### State Transitions & System Actions:
1. **DRAFT:** The customer constructs their cart. No database transaction is initialized yet.
2. **PENDING:** The order is submitted to the database. The system calculates taxes, updates the status to `'PENDING'`, and holds the order until payment is validated.
3. **ACCEPTED:** Payment is processed successfully. The order status updates to `'ACCEPTED'`, a kitchen queue card is generated, and a WebSocket event is broadcast to the branch's KDS displays.
4. **PREPARING:** Kitchen staff accept the ticket on a KDS screen. The status updates to `'PREPARING'`, starting the preparation timer.
5. **READY:** Cooking is completed. The status updates to `'READY'`, a notification is sent to the customer (via SMS or push notification), and the cashier terminal is alerted.
6. **COMPLETED:** The order is served, picked up, or delivered. The status updates to `'COMPLETED'`, final accounting invoice records are generated, and loyalty points are credited to the customer's account.
7. **CANCELLED:** If payment fails or a manager cancels the order, the status updates to `'CANCELLED'`. Any holds on inventory are released and returned to stock.

## 4.5 Real-Time Kitchen Dispatching (KDS) & Queue Management
The Kitchen Display System (KDS) manages kitchen operations and ticket flow in real-time.

- **WebSocket Synchronization:** The KDS communicates with the backend via high-speed, secure WebSockets (Socket.io). 
- **Room Scoping Isolation:** When a KDS terminal connects, it is placed in a WebSocket room scoped to its specific branch: `tenant_id:branch_id:kds`.
- **Dynamic Action Dispatching:** When an order is accepted, the order engine broadcasts a `ticket.created` event containing the order's items, options, and special notes to the branch's KDS room, updating KDS screens instantly.
- **Priority Escalation Logic:** The system monitors preparation times. If a ticket exceeds the product's estimated preparation time, its priority is escalated to `'RUSH'` on KDS screens, changing its border color to alert kitchen staff.

## 4.6 Cryptographically Secure QR Ordering Workflow
To prevent fraudulent ordering, table-specific QR codes are protected by dynamic cryptographic signatures.

- **Dynamic Token Generation:** Each table record contains a unique `qr_code_token` generated using a secure SHA-256 HMAC hash that signs the combination of the `tenant_id`, `branch_id`, `table_number`, and a rotating system pepper key:
  $$\text{Token} = \text{HMAC-SHA256}(\text{TenantID} + \text{BranchID} + \text{TableNum}, \text{SystemPepper})$$
- **Request Verification:** When a customer scans a table's QR code and submits an order, the backend validates the table ID and verifies that the provided token matches the database records. Orders with mismatched or missing table tokens are rejected. This prevents users from placing orders for random tables from outside the venue.

## 4.7 Subscription Gating & Usage-Based Billing Cycles
The system enforces subscription plan limits dynamically on every administrative request.

- **Middleware Enforcement:** Before executing mutating actions in the backoffice (such as adding branches or products), authorization middleware queries the tenant's current subscription status and plan limits.
- **Action Blocking:** If the tenant has reached their plan's limits, the middleware blocks the operation and returns an HTTP 403 Forbidden response with a descriptive error message (e.g., `'Branch limit reached. Please upgrade to a higher tier.'`).
- **Billing Sync Automation:** Subscriptions and cancellations are kept in sync via Stripe webhooks. When a webhook event is received, the billing service updates the tenant's subscription status in the database, updating their access permissions instantly.

## 4.8 Role-Based Permission Matrix Execution Path
Access control permissions are checked sequentially on every request to protect system resources.

1. **Access Token Extraction:** The gateway extracts the bearer JWT token from the incoming request's `Authorization` header.
2. **Signature Verification:** The gateway decrypts and verifies the token's cryptographic signature using the system's public key.
3. **Payload Verification:** The gateway parses the token's payload to extract user identity attributes, active roles, and assigned permissions.
4. **Tenant Scope Verification:** The system verifies that the target resource's `tenant_id` matches the user's tenant context.
5. **Guard Enforcement:** A custom NestJS permission guard evaluates the controller's required permissions against the user's active permissions. If the user lacks the necessary permissions, the request is rejected with an HTTP 403 Forbidden response and logged in the audit logging pipeline.

# 5. SECURITY IMPLEMENTATION SPECIFICATION

## 5.1 JWT Structure & Cryptographic Signing
Authentication is stateless and uses high-security JSON Web Tokens signed with an asymmetric **RS256** algorithm.
- **Asymmetric Encryption Key Management:** The authentication service generates a 2048-bit RSA key pair. The private key is stored securely in AWS Secrets Manager and used exclusively for signing tokens. The public key is distributed across microservices and gatekeepers, enabling them to verify tokens locally without hitting the central authentication database.
- **Access Token Payload Fields:**
  ```json
  {
    "sub": "u410c-9a1b-42b8-bf83-097a18fcd34c",
    "tenantId": "7a18f-39b0-4050-bf83-097a18fcd34b",
    "branchId": "b020c-9a1b-42b8-bf83-097a18fcd342",
    "email": "owner@restaurantbrand.com",
    "roles": ["RESTAURANT_OWNER"],
    "permissions": ["menu:create", "menu:update", "orders:read"],
    "iat": 1787161920,
    "exp": 1787162820,
    "iss": "https://auth.zayjar.com"
  }
  ```

## 5.2 Session Management & Sliding Window Revocation
The platform uses a hybrid session management approach to support fast stateless token validation while allowing for immediate session revocation when needed:
- **Stateless Token Verification:** API endpoints verify the asymmetric signature of incoming access tokens locally, avoiding database roundtrips.
- **Blacklist Revocation Engine:** Active revocations (such as user logouts, password resets, or security events) are tracked in a high-speed Redis cluster. When an access token is revoked, its unique ID (`jti`) is written to the Redis blacklist with a TTL matching the token's remaining lifespan. The API Gateway queries the Redis blacklist on every incoming request to reject blacklisted tokens.
- **Sliding Window Refresh Tokens:** Refresh tokens are tracked in both PostgreSQL and Redis. Every time a refresh token is used to renew an access token, the system rotates the refresh token: it invalidates the used token, generates a new one with a sliding-window expiration, and returns it to the client in a secure, HttpOnly cookie.

## 5.3 Cross-Site Request Forgery (CSRF) Mitigation
The API protects against Cross-Site Request Forgery (CSRF) attacks with robust security policies:
- **HttpOnly Cookie Attributes:** Refresh tokens are stored in cookies configured with strict security flags: `HostOnly; Secure; SameSite=Strict; Path=/; Domain=.zayjar.com`.
- **Double-Submit Cookie Patterns:** For administrative and user actions, the API Gateway requires the client to include a custom header `X-CSRF-Token` on all mutating requests (`POST`, `PUT`, `DELETE`). The gateway validates that the header value matches the CSRF token stored in the user's active session, rejecting requests with mismatched or missing tokens.

## 5.4 Cross-Site Scripting (XSS) Sanitization Pipelines
The platform applies comprehensive sanitization measures to prevent Cross-Site Scripting (XSS) vulnerabilities:
- **Strict Content Security Policy (CSP):** The web portals enforce strict Content Security Policy headers, restricting resource loading to trusted domains and blocking inline scripts, style injections, and unauthorized object executions:
  ```http
  Content-Security-Policy: default-src 'self'; script-src 'self' https://js.stripe.com; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; img-src 'self' data: https://cdn.zayjar.com; frame-src 'self' https://js.stripe.com;
  ```
- **Input Sanitization Filters:** Global NestJS middleware processes all incoming JSON payloads. It uses the `dompurify` and `class-sanitizer` libraries to sanitize HTML characters and strip malicious scripts before storing data in the database.

## 5.5 SQL Injection Defenses & ORM Boundaries
The system prevents SQL Injection attacks by restricting database access and routing all queries through an Object-Relational Mapping (ORM) boundary:
- **Parameterized Queries:** Relational operations are executed using **Prisma ORM**. Prisma handles database parameterization automatically, ensuring user inputs are treated as literal values rather than executable code.
- **Raw SQL Queries Restrictions:** The use of raw SQL queries is restricted. In rare cases where raw SQL is required for advanced reporting, developers must use Prisma's safe parameterized query methods (such as `prisma.$queryRaw`) to protect against injection vulnerabilities.

## 5.6 Distributed Rate Limiting via Redis Token Bucket
To protect against brute-force attacks and resource exhaustion, the API Gateway enforces rate limiting using a Redis-backed token bucket algorithm:
- **Rate Limit Tiers:**
  - Standard Public Endpoints (e.g., browsing menus): 120 requests per minute per IP.
  - Checkout & Order Submissions: 30 requests per minute per IP.
  - Authentication and Login Endpoints: 10 requests per minute per IP.
- **Redis Implementation:** The gateway tracks request limits in a Redis cache cluster. If a client exceeds their rate limit, the gateway blocks the request and returns an HTTP 429 Too Many Requests response with a standard retry-after header.

## 5.7 Immutable Structured Audit Logs
To support security compliance, tracking, and auditing, critical system operations write immutable records to the `audit_logs` table:
- **Log Archival Pipeline:** Every administrative mutation (including product updates, order cancellations, and configuration changes) triggers a background worker that writes detailed change records to the `audit_logs` table.
- **Immutable Storage:** The `audit_logs` table is configured with write-only permissions: records can be inserted, but `UPDATE` and `DELETE` operations are blocked at the database level. For long-term security, audit logs are periodically streamed to an external write-once-read-many (WORM) storage system.

## 5.8 Encryption standards (Transit, Rest, Application Level)
The platform applies industry-standard encryption standards to secure data across its lifecycle:
- **Data in Transit:** All network traffic is encrypted using TLS 1.3. Unencrypted HTTP requests are automatically redirected to HTTPS at the load balancer.
- **Data at Rest:** Persistent storage is encrypted at rest. AWS RDS database volumes, Redis caches, and S3 asset buckets use AES-256 encryption managed by AWS Key Management Service (KMS).
- **Application-Level Hashing:** User passwords are encrypted using the **Argon2id** hashing algorithm, protecting credentials against offline brute-force attacks.

## 5.9 Enterprise Secrets Management
To maintain system security, sensitive variables and secrets are managed through a centralized secret store:
- **Centralized Secrets Store:** Secrets (including API keys, private signing keys, database passwords, and provider credentials) are stored securely in **AWS Secrets Manager**.
- **Dynamic Injections:** Application containers fetch and inject required credentials dynamically into memory at startup, ensuring sensitive credentials are never stored or hardcoded in source repositories.

# 6. IMAGE STORAGE, PROCESSING, & OPTIMIZATION PIPELINE

## 6.1 Direct S3 Pre-Signed Upload Flow
To optimize server performance, the platform routes file uploads directly to AWS S3 using secure pre-signed URLs, bypassing the core application server.

```
[ Client ] --( requests pre-signed URL )-----> [ Application API ]
                                                      |
                                                      v
[ Client ] <--( returns pre-signed S3 URL )-- [ Validate session ]
    |
    v
[ Uploads file binary directly to S3 ]
    |
    v
[ S3 triggers Lambda to optimize image and save webp format ]
```

### Upload Execution Steps:
1. **Pre-signed URL Request:** The client sends an authorization request to the core API (`POST /api/v1/assets/presigned-url`) specifying the file's target content type, file size, and destination directory.
2. **Access Control Check:** The API validates that the user is authorized to upload assets. It checks that the file size is within limits (e.g., max 5MB for product photos) and matches allowed MIME types (`image/jpeg`, `image/png`, `image/webp`).
3. **URL Generation:** The API requests a secure pre-signed URL from AWS S3 using restricted IAM credentials. The URL is configured with a short expiration TTL (exactly 5 minutes) and is returned to the client.
4. **Direct Binary Upload:** The client uploads the image binary directly to S3 using the pre-signed URL via an HTTP PUT request.

## 6.3 Real-Time Serverless Image Optimization Engine
When an image is uploaded to S3, a serverless lambda worker processes and optimizes the file automatically:

- **S3 Upload Trigger:** S3 triggers an AWS Lambda worker when a new file is uploaded to the raw staging directory.
- **Image Processing Optimization:** The Lambda worker downloads the raw image and processes it using the high-performance **Sharp** image processing library:
  - Converts the image format to **WebP** to maximize compression.
  - Resizes the image to standardized dimensions depending on the image type (e.g., logo resized to 512x512, product photos resized to 1024x768).
  - Optimizes compression settings, targeting a quality level of 80% to balance file size and visual clarity.
- **Optimized Storage:** The Lambda worker writes the optimized WebP image back to the production S3 directory and deletes the raw staging image to clean up storage.

To speed up asset delivery worldwide, the platform caches optimized images on edge servers using AWS CloudFront:

- **Query Parameter Cache Invalidation:** To update cached images without manual invalidations, the platform uses a query parameter cache-busting strategy. When an image is updated, the database updates the image's timestamp, and the client loads the image using a cache-busting URL parameter (e.g., `/product_uuid.webp?v=1787161920`), forcing CloudFront to fetch and cache the updated asset.

# 9. PERFORMANCE OPTIMIZATION & HORIZONTAL SCALING

## 9.1 Redis Cache-Aside & Write-Through Caching Topologies
The platform uses high-speed Redis caches to minimize database load and speed up response times:
- **Cache-Aside Pattern:** High-read, low-write assets (such as menu items and categories) are cached using a Cache-Aside pattern:
  - When a client requests a menu, the application checks for the cached menu in Redis first.
  - If the menu is cached, it is returned immediately.
  - If there is a cache miss, the application queries the PostgreSQL database, writes the result to Redis, and returns the menu to the client.
- **Cache Invalidation Policies:** When a restaurant manager modifies their menu, the menu service invalidates the cached menu in Redis instantly. To prevent stale data, all cached keys are configured with a default time-to-live expiration (exactly 2 hours).

---

## 9.2 Advanced Database Optimization (Query Plans, Vacuuming, Index Maintenance)
To maintain quick query speeds as database tables grow, the system applies continuous database tuning:
- **Query Plan Profiling:** Database migrations and complex query filters are analyzed using PostgreSQL `EXPLAIN ANALYZE` tools to identify slow queries and optimize index design.
- **Automated Vacuuming:** Postgres `autovacuum` parameters are configured to regularly clean up dead rows, prevent table bloat, and update table statistics.
- **Database Connection Pooling:** **PgBouncer** connection pools are deployed in front of database replicas, maximizing concurrent connection limits and reducing connection initialization overhead.

---

## 9.3 Client-Side Performance & Bundle Optimizations
To ensure fast load times, particularly on mobile devices scanning QR codes, frontend applications apply modern performance optimizations:
- **Next.js Image Component:** Dynamic images are loaded using the Next.js `<Image />` component. Next.js optimizes images dynamically, converting them to WebP format, resizing them, and applying lazy loading automatically to speed up page loads.
- **Dynamic Imports & Code Splitting:** Heavy administrative modules and non-critical components are loaded dynamically on-demand using Next.js `dynamic()` imports, reducing the initial JavaScript bundle size.

---

## 9.4 Asynchronous Worker Architecture (BullMQ, Redis Streams)
The system uses a distributed background worker pool to handle non-blocking, resource-intensive tasks:
- **Distributed Queues:** Background processes (such as email dispatches, SMS notifications, image optimization, and webhook deliveries) are queued and processed asynchronously using **BullMQ** on a Redis database.
- **Failure Recovery:** Failed jobs are automatically retried up to 3 times using exponential backoff. If a job repeatedly fails, it is moved to a Dead Letter Queue (DLQ) for analysis and debugging.

---

## 9.5 Kubernetes-Driven Horizontal Pod Autoscaling (HPA)
The application container cluster scales dynamically to handle sudden spikes in traffic:
- **Horizontal Pod Autoscaling:** Container clusters are managed using Kubernetes or AWS ECS on Fargate. Dynamic autoscaling metrics are configured to scale the cluster out or in automatically based on application resource usage:
  - Scaling Out: The cluster launches additional container instances when average CPU usage exceeds 70% or average memory usage exceeds 80%.
  - Scaling In: Once usage falls below the thresholds, unnecessary containers are safely terminated to optimize infrastructure costs.

## B.1 Multi-Tenant Context Middleware

The following middleware extracts the tenant ID from the subdomain or custom domain on every incoming request, validates it against a Redis cache, and sets it in a thread-local async execution context.

```typescript
import { Injectable, NestMiddleware, HttpStatus, HttpException } from '@nestjs/common';
import { Request, Response, NextFunction } from 'express';
import { createClient } from 'redis';
import { AsyncLocalStorage } from 'async_local_storage';
import { PrismaService } from '../prisma/prisma.service';

// Instantiating thread-safe context storage
export const tenantContextStorage = new AsyncLocalStorage<{ tenantId: string }>();

@Injectable()
export class TenantContextMiddleware implements NestMiddleware {
  private redisClient;

  constructor(private readonly prisma: PrismaService) {
    // Initializing high-speed caching redis connection
    this.redisClient = createClient({ url: process.env.REDIS_URL });
    this.redisClient.connect().catch(console.error);
  }

  async use(req: Request, res: Response, next: NextFunction) {
    const host = req.headers.host || '';
    const correlationId = req.headers['x-correlation-id'] || `req-${Math.random().toString(36).substr(2, 9)}`;
    req.headers['x-correlation-id'] = correlationId;

    let tenantId: string | null = null;

    try {
      // 1. Check for manual override header (authorized developer scoping)
      const headerTenantId = req.headers['x-tenant-id'];
      if (headerTenantId && typeof headerTenantId === 'string') {
        tenantId = headerTenantId;
      } else {
        // 2. Extract subdomain or custom domain mapping
        const parsedHost = host.split(':')[0]; // Strip port if present
        
        if (parsedHost.endsWith('.zayjar.com')) {
          const subdomain = parsedHost.replace('.zayjar.com', '');
          // Resolve subdomain from cache first
          tenantId = await this.redisClient.get(`subdomain:${subdomain}`);
          
          if (!tenantId) {
            // Cache miss: query PostgreSQL database
            const tenant = await this.prisma.tenant.findUnique({
              where: { subdomain, deletedAt: null },
              select: { id: true }
            });
            if (tenant) {
              tenantId = tenant.id;
              // Write back to cache with 1-hour expiration
              await this.redisClient.set(`subdomain:${subdomain}`, tenantId, { EX: 3600 });
            }
          }
        } else {
          // Resolve verified custom domain mapping
          tenantId = await this.redisClient.get(`domain:${parsedHost}`);
          
          if (!tenantId) {
            const tenant = await this.prisma.tenant.findUnique({
              where: { customDomain: parsedHost, deletedAt: null },
              select: { id: true }
            });
            if (tenant) {
              tenantId = tenant.id;
              await this.redisClient.set(`domain:${parsedHost}`, tenantId, { EX: 3600 });
            }
          }
        }
      }

      if (!tenantId) {
        throw new HttpException(
          {
            statusCode: HttpStatus.NOT_FOUND,
            timestamp: new Date().toISOString(),
            path: req.originalUrl,
            error: 'Tenant Not Found',
            message: 'Unable to resolve the active tenant domain context.',
            correlationId
          },
          HttpStatus.NOT_FOUND
        );
      }

      // 3. Inject resolved tenant context into thread local storage
      tenantContextStorage.run({ tenantId }, () => {
        // Bind dynamic properties to the request context
        req['tenantId'] = tenantId;
        next();
      });

    } catch (error) {
      if (error instanceof HttpException) {
        return res.status(error.getStatus()).json(error.getResponse());
      }
      return res.status(HttpStatus.INTERNAL_SERVER_ERROR).json({
        statusCode: HttpStatus.INTERNAL_SERVER_ERROR,
        timestamp: new Date().toISOString(),
        path: req.originalUrl,
        error: 'Internal Server Error',
        message: 'A fatal exception occurred during tenant domain scoping.',
        correlationId
      });
    }
  }
}
```

## B.2 Role-Based Access Control (RBAC) & Tenant Gating Guard

The following guard enforces role assignments and resource permissions, ensuring that users can only operate on data scoping to their authorized tenant workspace.

```typescript
import { Injectable, CanActivate, ExecutionContext, HttpStatus, HttpException } from '@nestjs/common';
import { Reflector } from '@nestjs/core';
import * as jwt from 'jsonwebtoken';

@Injectable()
export class RbacPermissionGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  async canActivate(context: ExecutionContext): Promise<boolean> {
    const request = context.switchToHttp().getRequest();
    const correlationId = request.headers['x-correlation-id'];

    // Retrieve handler-level metadata permissions tags
    const requiredPermission = this.reflector.get<string>('permission', context.getHandler());
    if (!requiredPermission) {
      return true; // Unprotected route
    }

    const authHeader = request.headers.authorization;
    if (!authHeader || !authHeader.startsWith('Bearer ')) {
      throw new HttpException(
        {
          statusCode: HttpStatus.UNAUTHORIZED,
          timestamp: new Date().toISOString(),
          path: request.originalUrl,
          error: 'Unauthorized',
          message: 'Access token was not provided or has expired.',
          correlationId
        },
        HttpStatus.UNAUTHORIZED
      );
    }

    const token = authHeader.split(' ')[1];
    
    try {
      // Validate signature against the RSA Public Key injected at runtime
      const publicKey = process.env.JWT_PUBLIC_KEY;
      const decodedPayload = jwt.verify(token, publicKey, { algorithms: ['RS256'] }) as any;
      
      request.user = decodedPayload;

      // Ensure user tenant context matches active domain tenant context
      if (decodedPayload.tenantId && decodedPayload.tenantId !== request.tenantId) {
        throw new HttpException(
          {
            statusCode: HttpStatus.FORBIDDEN,
            timestamp: new Date().toISOString(),
            path: request.originalUrl,
            error: 'Forbidden',
            message: 'Cross-tenant access attempt was detected and blocked.',
            correlationId
          },
          HttpStatus.FORBIDDEN
        );
      }

      // Check if user holds the platform owner administrator master key
      if (decodedPayload.roles.includes('PLATFORM_OWNER')) {
        return true;
      }

      // Evaluate permission string matching
      const hasPermission = decodedPayload.permissions.includes(requiredPermission);
      if (!hasPermission) {
        throw new HttpException(
          {
            statusCode: HttpStatus.FORBIDDEN,
            timestamp: new Date().toISOString(),
            path: request.originalUrl,
            error: 'Forbidden',
            message: `Your active credentials lack the required permission: [${requiredPermission}].`,
            correlationId
          },
          HttpStatus.FORBIDDEN
        );
      }

      return true;

    } catch (error) {
      if (error instanceof HttpException) {
        throw error;
      }
      throw new HttpException(
        {
          statusCode: HttpStatus.UNAUTHORIZED,
          timestamp: new Date().toISOString(),
          path: request.originalUrl,
          error: 'Unauthorized',
          message: 'The provided access token was invalid or has been blacklisted.',
          correlationId
        },
        HttpStatus.UNAUTHORIZED
      );
    }
  }
}
```

## B.3 Transactional Order Placement & Pricing Engine Service

The following NestJS service orchestrates order placement in a single database transaction, applying price inheritance rules, calculating taxes dynamically, updating table statuses, and broadcasting updates to kitchen displays.

```typescript
import { Injectable, HttpStatus, HttpException } from '@nestjs/common';
import { PrismaService } from '../prisma/prisma.service';
import { KdsGateway } from '../kds/kds.gateway';
import { CreateOrderDto } from './dto/create-order.dto';

@Injectable()
export class OrderService {
  constructor(
    private readonly prisma: PrismaService,
    private readonly kdsGateway: KdsGateway
  ) {}

  async checkout(tenantId: string, dto: CreateOrderDto): Promise<any> {
    const correlationId = `ord-tx-${Math.random().toString(36).substr(2, 9)}`;

    // Execute the complete order assembly inside a managed database transaction
    return this.prisma.$transaction(async (tx) => {
      
      // 1. Fetch branch configuration settings
      const branch = await tx.branch.findUnique({
        where: { id: dto.branchId, tenantId, deletedAt: null },
        include: { restaurant: true }
      });
      if (!branch) {
        throw new HttpException('Target branch was not found under this tenant profile.', HttpStatus.NOT_FOUND);
      }

      // 2. Validate table occupancy rules for DINE_IN checkouts
      if (dto.type === 'DINE_IN' && dto.tableId) {
        const table = await tx.table.findUnique({
          where: { id: dto.tableId, branchId: dto.branchId, deletedAt: null }
        });
        if (!table) {
          throw new HttpException('Target dining table does not exist in this branch.', HttpStatus.NOT_FOUND);
        }
        if (table.status === 'OCCUPIED') {
          throw new HttpException('The selected table is occupied.', HttpStatus.CONFLICT);
        }
        // Lock table occupancy state
        await tx.table.update({
          where: { id: dto.tableId },
          data: { status: 'OCCUPIED' }
        });
      }

      let orderSubtotal = 0;
      const orderItemsToCreate = [];

      // 3. Process shopping cart items sequentially
      for (const cartItem of dto.items) {
        const product = await tx.product.findUnique({
          where: { id: cartItem.productId, tenantId, deletedAt: null }
        });
        if (!product || !product.isAvailable) {
          throw new HttpException(`Product with ID [${cartItem.productId}] is currently unavailable.`, HttpStatus.UNPROCESSABLE_ENTITY);
        }

        let itemUnitPrice = Number(product.basePrice);

        // A. Evaluate sizing adjustments
        if (cartItem.sizeId) {
          const size = await tx.productSize.findUnique({
            where: { id: cartItem.sizeId, productId: product.id }
          });
          if (!size) {
            throw new HttpException(`Size modifier [${cartItem.sizeId}] is invalid for this product.`, HttpStatus.UNPROCESSABLE_ENTITY);
          }
          itemUnitPrice += Number(size.priceAdjustment);
        }

        // B. Evaluate variant price overrides
        if (cartItem.variantId) {
          const variant = await tx.productVariant.findUnique({
            where: { id: cartItem.variantId, productId: product.id }
          });
          if (!variant) {
            throw new HttpException(`Variant [${cartItem.variantId}] is invalid for this product.`, HttpStatus.UNPROCESSABLE_ENTITY);
          }
          if (variant.stockQuantity < cartItem.quantity) {
            throw new HttpException(`Insufficient inventory for variant: ${variant.name}.`, HttpStatus.CONFLICT);
          }
          itemUnitPrice = Number(variant.price); // Direct override

          // Decrement variant stock inventory
          await tx.productVariant.update({
            where: { id: cartItem.variantId },
            data: { stockQuantity: { decrement: cartItem.quantity } }
          });
        }

        let itemAddonsTotal = 0;
        const addonsToCreate = [];

        // C. Evaluate addon modifications
        if (cartItem.addons && cartItem.addons.length > 0) {
          for (const addonSelection of cartItem.addons) {
            const addonItem = await tx.addonItem.findUnique({
              where: { id: addonSelection.addonItemId, isAvailable: true }
            });
            if (!addonItem) {
              throw new HttpException(`Selected addon customization is unavailable.`, HttpStatus.UNPROCESSABLE_ENTITY);
            }
            itemAddonsTotal += Number(addonItem.price);
            addonsToCreate.push({
              tenantId,
              addonItemId: addonItem.id,
              price: addonItem.price
            });
          }
        }

        const calculatedLinePrice = (itemUnitPrice + itemAddonsTotal) * cartItem.quantity;
        orderSubtotal += calculatedLinePrice;

        orderItemsToCreate.push({
          tenantId,
          productId: product.id,
          sizeId: cartItem.sizeId || null,
          variantId: cartItem.variantId || null,
          quantity: cartItem.quantity,
          unitPrice: itemUnitPrice,
          totalPrice: calculatedLinePrice,
          cookingStatus: 'PENDING',
          orderItemAddons: {
            create: addonsToCreate
          }
        });
      }

      // 4. Run tax and financial ledger calculations
      const taxRate = Number(branch.restaurant.taxPercentage) / 100;
      const taxAmount = Number((orderSubtotal * taxRate).toFixed(2));
      const orderTotal = Number((orderSubtotal + taxAmount).toFixed(2));

      // 5. Generate human-friendly order reference number
      const sequenceValue = await tx.$queryRawUnsafe<any>(`SELECT nextval('order_seq_${branch.id}') as val`);
      const orderNumber = `ORD-${new Date().getFullYear()}-${sequenceValue[0].val.toString().padStart(5, '0')}`;

      // 6. Write final Order to database
      const order = await tx.order.create({
        data: {
          tenantId,
          branchId: dto.branchId,
          tableId: dto.tableId || null,
          orderNumber,
          type: dto.type,
          status: 'PENDING', // Awaiting payment verification
          subtotal: orderSubtotal,
          taxAmount,
          total: orderTotal,
          specialNotes: dto.specialNotes || null,
          orderItems: {
            create: orderItemsToCreate
          }
        },
        include: {
          orderItems: {
            include: {
              orderItemAddons: true
            }
          }
        }
      });

      // 7. Dispatch ticket to real-time Kitchen Displays
      if (dto.paymentMethod === 'CASH') {
        // Cash transactions go directly to KDS queue
        await tx.order.update({
          where: { id: order.id },
          data: { status: 'ACCEPTED' }
        });
        
        // Push notifications payload to branch screens
        this.kdsGateway.broadcastToBranch(dto.branchId, 'ticket.created', {
          ticketId: order.id,
          ticketNumber: orderNumber.substr(-3),
          priority: 'NORMAL',
          items: order.orderItems
        });
      }

      return {
        orderId: order.id,
        orderNumber,
        financials: {
          subtotal: orderSubtotal,
          taxAmount,
          total: orderTotal
        },
        status: order.status
      };
    });
  }
}
```

## C.2 Kitchen Display System (KDS) Queue Terminal Screen

The following React Component integrates directly with Socket.io real-time broadcast servers, receiving KDS ticketing signals and performing interactive priority ticket status rendering.

```typescript
'use client';

import React, { useEffect, useState } from 'react';
import io from 'socket.io-client';

interface OrderItem {
  id: string;
  name: string;
  quantity: number;
  size: string | null;
  addons: string[];
  cookingStatus: 'PENDING' | 'PREPARING' | 'COOKED';
}

interface KDSTicket {
  ticketId: string;
  ticketNumber: string;
  priority: 'LOW' | 'NORMAL' | 'RUSH';
  elapsedMinutes: number;
  items: OrderItem[];
}

export const KDSTerminal: React.FC<{ branchId: string; accessToken: string }> = ({ branchId, accessToken }) => {
  const [tickets, setTickets] = useState<KDSTicket[]>([]);

  useEffect(() => {
    // Handshake initialization with secure credentials validation
    const socket = io(process.env.NEXT_PUBLIC_SOCKET_SERVER_URL || '', {
      auth: { token: accessToken },
      query: { branchId }
    });

    socket.on('connect', () => {
      console.log('KDS WebSocket established successfully.');
    });

    // Handle new incoming tickets
    socket.on('ticket.created', (newTicket: KDSTicket) => {
      setTickets((prevTickets) => [...prevTickets, { ...newTicket, elapsedMinutes: 0 }]);
    });

    // Update active state mappings
    socket.on('ticket.item_updated', (updatePayload: { orderItemId: string; status: 'PREPARING' | 'COOKED' }) => {
      setTickets((prevTickets) =>
        prevTickets.map((ticket) => ({
          ...ticket,
          items: ticket.items.map((item) =>
            item.id === updatePayload.orderItemId
              ? { ...item, cookingStatus: updatePayload.status }
              : item
          )
        }))
      );
    });

    return () => {
      socket.disconnect();
    };
  }, [branchId, accessToken]);

  const updateItemStatus = async (orderItemId: string, currentStatus: string) => {
    const nextStatus = currentStatus === 'PENDING' ? 'PREPARING' : 'COOKED';
    
    // Optimistic client mutation
    setTickets((prevTickets) =>
      prevTickets.map((ticket) => ({
        ...ticket,
        items: ticket.items.map((item) =>
          item.id === orderItemId ? { ...item, cookingStatus: nextStatus as any } : item
        )
      }))
    );

    await fetch(`/api/v1/kds/items/${orderItemId}/status`, {
      method: 'PUT',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${accessToken}`
      },
      body: JSON.stringify({ status: nextStatus })
    });
  };

  return (
    <div className="w-full bg-slate-900 min-h-screen p-6 flex flex-col">
      <header className="flex justify-between items-center border-b border-slate-800 pb-4 mb-6">
        <h1 className="text-xl font-bold text-white">Active Production Board</h1>
        <div className="flex gap-4">
          <span className="text-xs text-slate-400">Connected</span>
          <span className="w-3 h-3 rounded-full bg-green-500 self-center"></span>
        </div>
      </header>

      {/* Grid of Kitchen Tickets */}
      <main className="flex-1 grid grid-cols-4 gap-6 items-start overflow-x-auto">
        {tickets.map((ticket) => (
          <div
            key={ticket.ticketId}
            className={`rounded-xl border shadow-xl flex flex-col max-h-[80vh] overflow-hidden ${
              ticket.priority === 'RUSH' ? 'border-red-500 bg-red-950/20' : 'border-slate-800 bg-slate-950'
            }`}
          >
            {/* Ticket Header card */}
            <div className="px-4 py-3 border-b border-slate-800 flex justify-between items-center">
              <span className="text-2xl font-black text-white">#{ticket.ticketNumber}</span>
              <span className={`text-[10px] px-2 py-1 rounded font-bold ${
                ticket.priority === 'RUSH' ? 'bg-red-500 text-white animate-pulse' : 'bg-slate-800 text-slate-400'
              }`}>
                {ticket.priority}
              </span>
            </div>

            {/* Ticket Order items */}
            <div className="p-4 flex-1 overflow-y-auto space-y-4">
              {ticket.items.map((item) => (
                <div key={item.id} className="border-b border-slate-800/50 pb-3 last:border-0">
                  <div className="flex justify-between">
                    <span className="text-sm font-bold text-slate-100">{item.quantity}x {item.name}</span>

                      onClick={() => updateItemStatus(item.id, item.cookingStatus)}
                      className={`text-[10px] px-2 py-1 rounded font-bold transition-colors ${
                        item.cookingStatus === 'COOKED'
                          ? 'bg-green-500 text-white'
                          : item.cookingStatus === 'PREPARING'
                          ? 'bg-yellow-500 text-black'
                          : 'bg-slate-700 text-slate-300 hover:bg-slate-600'
                      }`}
                    >
                      {item.cookingStatus}
                    </button>
                  </div>
                  {item.size && <p className="text-xs text-slate-400 mt-1">Size: {item.size}</p>}
                  {item.addons.length > 0 && (
                    <div className="flex flex-wrap gap-1 mt-2">
                      {item.addons.map((add, idx) => (
                        <span key={idx} className="bg-slate-800 text-slate-300 text-[10px] px-1.5 py-0.5 rounded">
                          +{add}
                        </span>
                      ))}
                    </div>
                  )}
                </div>
              ))}
            </div>
          </div>
        ))}
      </main>
    </div>
  );
};
```

# APPENDIX F: OPERATIONAL RUNBOOKS & TROUBLESHOOTING RUNBOOKS

This appendix provides production-grade operational runbooks, incident recovery steps, and diagnostic procedures for site reliability engineers (SREs) and platform operations teams managing the Zayjar Restaurant SaaS Platform.

## F.1 Database Failover & High-Availability Recovery Plan

In the event of a physical primary database node failure on AWS RDS, the following sequence details the automated failover execution path, proxy connection cycling, and read-replica re-routing.

```
       [ AWS RDS Primary Fail ]
                  |
                  v ( Heartbeat loss detected by Standby Node )
     [ Standby Node Promoted ]
                  |
                  +-----------------------+-----------------------+
                  |                                               |
                  v ( AWS DNS Updates )                           v ( PgBouncer Connection Reset )
        [ Read/Write Resumed ]                          [ Active Queries Flushed ]
```

### Incident Diagnostics & Failover Verification Sequence:
1. **Heartbeat Loss Detection:** The AWS RDS Multi-AZ manager registers heartbeat loss on the Primary database instance (e.g., node timeout or storage volume lockup).
2. **Standby Node Promotion:** The standby RDS node is promoted to become the new Primary Write instance. 
3. **AWS DNS Routing Update:** AWS updates the virtual IP address mapped to the main RDS writer connection endpoint. This update usually propagates within 30 to 60 seconds.
4. **PgBouncer Connection Cycling:** During the database promotion, active database queries on application containers will throw socket timeout exceptions (such as `P2025` or transaction execution timeouts). To clear dead socket connections immediately:
   - SREs or automated scaling agents execute a connection flush command on PgBouncer nodes:
     ```bash
     pgbouncer -u pgbouncer_admin -p SecretPgbouncerAdmin123! -c "RECONNECT"
     ```
   - This command drops all cached sockets and forces PgBouncer to establish new, clean socket connections to the newly promoted RDS writer endpoint.
5. **Secondary Verification checks:** SREs execute verification checks on the core database cluster to ensure read and write operations have resumed successfully:
   ```sql
   -- Verify write operations
   INSERT INTO audit_logs (action, entity_name, ip_address, user_agent)
   VALUES ('FAILOVER_DIAGNOSTIC_WRITE', 'RDS_HEALTH_CHECK', '127.0.0.1', 'AWS_HEALTH_MONITOR');
   
   -- Verify replica replication lag
   SELECT pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS replication_lag_bytes
   FROM pg_stat_replication;
   ```

## F.2 Cache Invalidation Failures & Redis Desynchronization Runbook

In rare operational scenarios where Redis cache entries fall out of sync with the persistent PostgreSQL database (for example, if a background worker fails or a network partition blocks cache invalidation events), customer menus may display stale menu items or incorrect prices.

### Diagnostic Steps & Runbook Remedies:
1. **Locating Cache Desynchronization:** To confirm a cache desynchronization event, SREs compare the target tenant's product properties in PostgreSQL directly against the serialized string stored in Redis:
   ```bash
   # Read cached product object from Redis
   redis-cli -u redis://:SecretRedis123!@redis-cache:6379 GET "tenant:7a18:product:p222"
   ```
   If the Redis JSON output displays outdated base prices compared to the database, a cache invalidation failure has occurred.
2. **Targeted Cache Invalidation:** To restore service immediately without affecting other tenant caches, SREs delete the specific tenant's stale menu cache key in Redis:
   ```bash
   # Remove targeted cache key
   redis-cli -u redis://:SecretRedis123!@redis-cache:6379 DEL "tenant:7a18:menu"
   ```
   Subsequent client requests to the customer QR menu will result in a cache miss, prompting the application server to query PostgreSQL and cache the updated menu data.
3. **Emergency Cache Flush (Cluster-Wide):** If a system-wide database update fails to invalidate caches across the entire network, SREs can execute a cluster-wide cache flush:
   ```bash
   # Flush cached query entries across all cluster nodes safely
   redis-cli -u redis://:SecretRedis123!@redis-cache:6379 FLUSHALL ASYNC
   ```
   *Warning:* Flush operations must use the `ASYNC` modifier to perform the cleanup in the background, preventing Redis from blocking and timing out active WebSocket and queue services.

## F.3 Multi-Tenant Outbound Webhook Queue Backlog Incident Response

If a tenant's third-party integration server goes offline, outbound webhook dispatch queues may build up in Redis, consuming system memory and slowing down background notifications.

### Action Plan & Remediation Sequence:
1. **Identify Backlog Volume:** SREs log into the Redis console and query the size of active and failed BullMQ webhook queues:
   ```bash
   # Check active queue lengths
   redis-cli -u redis://:SecretRedis123!@redis-cache:6379 LLEN "bull:webhooks:active"
   redis-cli -u redis://:SecretRedis123!@redis-cache:6379 LLEN "bull:webhooks:wait"
   ```
2. **Identify Failing Endpoints:** Review Winston logs to pinpoint the specific `tenant_id` and webhook target URL that is causing connection timeouts:
   ```bash
   # Parse application logs for webhook failure warnings
   grep "WEBHOOK_DISPATCH_FAILED" /var/log/zayjar-api/error.log | jq '.metadata' | sort | uniq -c
   ```
3. **Enforce Temporary Webhook Pause:** If a specific tenant's integration server remains offline and causes a backlog of over 50,000 failed webhook requests, SREs temporarily pause the tenant's webhook subscription to protect system resources:
   ```sql
   -- Pause the failing tenant's webhook in PostgreSQL
   UPDATE webhooks 
   SET is_active = FALSE 
   WHERE tenant_id = '7a18f-39b0-4050-bf83-097a18fcd34b' AND target_url = 'https://broken-partner.com/callback';
   ```
4. **Purging Failed Backlog Jobs:** SREs clear out failed webhook jobs from the active queue to restore background processing speeds:
   ```bash
   # Safe background cleanup of waiting job lists
   redis-cli -u redis://:SecretRedis123!@redis-cache:6379 DEL "bull:webhooks:wait"
   ```
   Once the tenant's integration server is back online, the webhook configuration can be reactivated, and subsequent event dispatches will resume automatically.

# APPENDIX G: REGULATORY COMPLIANCE, VAT TAXATION, & DATA RETENTION

This appendix details the technical implementation guidelines for localized tax calculations, fiscal invoice reporting, electronic archiving, and privacy compliance on the Zayjar Restaurant SaaS Platform.

## G.1 Localized VAT Taxation & Fiscal Integrity Engine

In many operating jurisdictions (including the European Union and the Gulf Cooperation Council - GCC), SaaS restaurant systems must adhere to strict electronic invoicing (e-Invoicing) regulations and tax structures:
- **Saudi Arabia ZATCA Compliance:** For the Saudi market, the platform's invoicing service integrates with the **Zakat, Tax and Customs Authority (ZATCA)** APIs.
- **Dynamic XML Signature Generation:** For every completed checkout transaction, the system generates an XML invoice conforming to UBL 2.1 standard schemas. The invoice metadata (including timestamps, subtotal, VAT tax rate, and final totals) is cryptographically signed using a private certificate issued during tenant onboarding:
  $$\text{Invoice Hash} = \text{SHA256}(\text{UBL XML XML Canonicalized Payload})$$
- **Secure QR Code Generation:** The generated invoice must feature an electronic QR code representing specific serialized TLV (Tag-Length-Value) attributes:
  - Tag 1: Seller Brand Name.
  - Tag 2: Seller VAT Registration Number.
  - Tag 3: Transaction Timestamp.
  - Tag 4: Invoice Total (with VAT).
  - Tag 5: VAT Tax Total.
- **Base64 Packaging:** The compiled TLV array is Base64 encoded and rendered as a high-density, secure QR code on customer-facing digital receipts and physical paper slips.

## G.2 Privacy & General Data Protection Regulation (GDPR) Guardrails

To operate legally within European markets, the platform implements technical controls to enforce strict data privacy standards:
- **Right to Be Forgotten (Erasure Engine):** When a customer requests account erasure, the system processes a cascading data scrubbing transaction:
  - Email, first name, last name, and telephone columns are wiped from the `customers` record or overwritten with randomized placeholder values (such as `anonymized-customer-uuid@zayjar.com`).
  - Active session hashes and push device tokens mapped to the customer are deleted instantly from the `session_logs` and `device_tokens` tables.
  - *Data Retention Overrides:* To comply with fiscal tax reporting audits, relational order line items and tax invoice receipts are preserved; however, they are stripped of all personal identifying information (PII).
- **Consent Tracking Logging:** User consent events (such as terms-of-service and marketing opt-ins) are written to the `audit_logs` database table with timestamp and IP signatures, establishing an immutable compliance audit trail.

## G.3 Structured Data Retention & Deletion Policy

To optimize PostgreSQL database sizing and prevent disk space bloat, the system enforces a strict data lifecycle retention schedule:

| Table Name | Active Retention Window | Archiving Destination | Purge Trigger Event |
| --- | --- | --- | --- |
| `audit_logs` | 180 Days (Active) | AWS S3 Glacier WORM Storage | Auto Purged after 7 Years |
| `session_logs` | 30 Days (Active) | Direct Purged in Postgres | Auto deleted on expiration |
| `notifications` | 90 Days (Active) | Direct Purged in Postgres | Auto cleared by daily cron task |
| `orders` | 3 Years (Active) | Parquet compressed on AWS S3 | Logical soft-deletes preserved |
| `invoices` | 10 Years (Auditable) | AWS S3 Standard Archive | Locked under fiscal retention rules |

### Automated Cleanup Cron Job Definition:
At 02:00 UTC daily, a serverless cleanup task executes database partitions rotation and retention sweeping:
```sql
-- Clean up expired user sessions from PostgreSQL
DELETE FROM session_logs 
WHERE expires_at < NOW() - INTERVAL '30 days';

-- Purge read notifications older than 90 days
DELETE FROM notifications 
WHERE is_read = TRUE AND created_at < NOW() - INTERVAL '90 days';
```
This ensures high-availability relational indexes remain compact and query execution latency averages sub-millisecond rates across high-frequency tables.

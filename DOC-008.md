# 7. NOTIFICATIONS SYSTEM ARCHITECTURE

## 7.1 Multi-Channel Dispatch Engine
The notifications system uses an asynchronous multi-channel dispatch engine to deliver real-time updates across several channels:

```
                          [ Dispatch Event Triggered ]
                                       |
                   +-------------------+-------------------+
                   |                   |                   |
                   v                   v                   v
            [ Email Channel ]    [ SMS Channel ]     [ Push Channel ]
            - SendGrid SES       - Twilio API        - Firebase FCM
```

- **Asynchronous Processing:** To prevent blocking core API requests, notification dispatches are offloaded to an asynchronous queue system powered by **BullMQ** and **Redis Streams**.
- **Failover Routing:** If a primary delivery provider fails (such as an API outage on the primary SMS gateway), the notification worker catches the error, marks the attempt as failed, and automatically routes the dispatch to a configured secondary provider.

---

## 7.2 Transactional & Marketing Email Pipelines
The platform integrates with AWS SES or SendGrid to manage and deliver transactional and marketing emails:
- **Template System:** Email layouts (such as signup confirmations, invoice receipts, and password resets) are designed using Handlebars and compiled dynamically with user variables before dispatch.
- **Delivery Monitoring:** The email worker monitors delivery events (such as bounces and spam complaints) via webhook feedback loops. Hard bounces are flagged in the database, and subsequent dispatches to those addresses are blocked automatically to protect sender reputation.

---

## 7.3 SMS Delivery Routing & Fault Tolerance
Transactional notifications (such as order status updates and OTP logins) are delivered via SMS through regional integrations:
- **SMS Gateway Integration:** The system routes international messages through the **Twilio SMS API**.
- **Regional Route Optimization:** For Middle Eastern regions, the system integrates with specialized regional SMS gateways (such as **Unifonic** or **SAPTCO SMS**) to optimize delivery speeds and minimize message costs.
- **Failover Queue:** In the event of gateway timeouts or routing failures, the worker automatically queues the message for redelivery using a secondary route.

---

## 7.4 Firebase Cloud Messaging (FCM) Integration
Real-time administrative alerts and staff notifications are delivered using Firebase Cloud Messaging (FCM):
- **Token Registration:** Administrative and staff devices register their push subscription tokens with the system, storing them in the `device_tokens` table.
- **Payload Structure:** Notification workers construct and send targeted payloads containing the message title, body, and routing metadata:
  ```json
  {
    "message": {
      "token": "fcm_token_val_123...",
      "notification": {
        "title": "New Order Placed",
        "body": "Order ORD-2026-10045 was successfully submitted."
      },
      "data": {
        "orderId": "o888c-9a1b-42b8-bf83-097a18fcd341",
        "action": "view_order"
      }
    }
  }
  ```

## 7.5 Outbound Webhook Subsystem
Premium subscription plans can configure custom webhooks to receive real-time updates on critical order events:
- **Subscribed Events:** External developer systems can register to receive webhook events for events such as `order.created`, `order.accepted`, or `order.completed`.
- **Payload Signature Signing:** To secure webhooks and enable verification, the webhook service signs outgoing payloads using an HMAC-SHA256 signature generated with the tenant's private webhook secret. The signature is sent in the `X-Zayjar-Signature` request header.
- **Automatic Retry Policy:** If a tenant's server returns a non-2xx response status, the webhook worker initiates a retry sequence using exponential backoff (retrying up to 5 times over a 24-hour period before pausing the webhook endpoint).

---

## 7.6 Real-Time In-App Notifications & WebSocket Synchronization
In-app notifications and KDS tickets are kept in sync in real-time using secure WebSocket connections:
- **Socket.io Servers:** The system uses high-performance Node.js Socket.io servers backed by Redis adapters to manage active connections across application instances.
- **Connection Authentication:** Clients authenticate WebSocket connections by passing a valid JWT bearer token in the connection handshake. Connections with invalid or missing tokens are rejected.
- **Event Broadcasts:** When order statuses are updated, the backend broadcasts websocket events to the specific room scoped to the active branch (`tenant_id:branch_id`), triggering instant UI updates on active client applications.

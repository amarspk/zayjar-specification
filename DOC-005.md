# 4. BUSINESS LOGIC & WORKFLOWS

## 4.1 Tenant & Restaurant Onboarding Workflow
When a new restaurant owner signs up for the Zayjar Restaurant SaaS Platform, the system initiates a structured onboarding process to provision their tenant environment and prepare it for use.

```
[ Signup Requested ] 
        |
        v
[ Validate unique subdomain and email ] 
        |
        v
[ DB Transaction Block Init ] 
  ├── Insert tenant profile
  ├── Create default Subscription (TRIALING)
  ├── Register User account (RESTAURANT_OWNER)
  ├── Initialize default Restaurant Brand
  └── Create first standard Branch (Main Office)
        |
        v
[ Provisions Stripe Customer Profile ] 
        |
        v
[ Deliver Welcome Email Sequence ]
```

### Onboarding Steps:
1. **Validation Checks:** The application validates that the requested subdomain is available, conformant to naming standards, and not on the reserved subdomains list (e.g., `'admin'`, `'api'`, `'www'`). It also checks that the registered email address is unique.
2. **Database Transaction Block:** To ensure database consistency, all onboarding operations are wrapped inside a database transaction:
   - **Create Tenant:** A new `tenants` record is initialized with the company name, dynamic subdomain, and default branding configurations.
   - **Create Subscription:** A `subscriptions` record is generated, linking the tenant to the trial plan and setting a trial end date 14 days out.
   - **Register Owner Account:** A `users` record is initialized with the owner's details and cryptographically hashed password. The user is assigned the `'RESTAURANT_OWNER'` role.
   - **Create Restaurant Entity:** A `restaurants` record is generated, setting default values for taxes, currency, and timezone based on the tenant's location.
   - **Create Default Branch:** A `branches` record is generated to serve as the brand's primary branch location, pre-populated with standard operating hours templates.
3. **Stripe Profile Provisioning:** The tenant is registered in Stripe, creating a customer profile and linking its ID (`stripe_customer_id`) back to the tenant record.
4. **Welcome Sequence Dispatch:** A background worker sends a transactional welcome email sequence via SendGrid containing setup instructions and guides.

## 4.2 Multi-Branch Data Scoping & Switching Workflow
The system is built to support multi-branch operations, enabling a single tenant to manage several branches under one brand.

```
                              [ API Gateways Routing ]
                                         |
                       +-----------------+-----------------+
                       |                                   |
                       v                                   v
             [ Customer Scanning ]               [ Backoffice / Staff Login ]
             - Fixed Branch Scoping              - Dynamic JWT token context
             - Read Only (Menu)                  - Branch switcher UI trigger
                                                 - Select and scope context
```

- **Authentication Scope Gating:** When an administrative user or staff member logs in, their access token includes a list of branches they are assigned to.
- **Dynamic Context Switcher:** When staff switch locations in the backoffice UI, the application client updates the `X-Branch-ID` request header with the selected branch UUID.
- **Dynamic Scoping Interceptor:** On the backend, a scoping interceptor validates the `X-Branch-ID` header value. It ensures the user has access to that location and restricts all subsequent database queries to data matching that specific branch ID.

## 4.3 Menu Item Price Inheritance Engine
The menu system supports flexible pricing configurations, allowing restaurants to set base prices that can be overridden at the size or variant level.

```
                              [ Evaluate Product Price ]
                                          |
                      +-------------------+-------------------+
                      |                                       |
            [ Variant Selected? ]                     [ Size Selected? ]
                      |                                       |
             (Variant Price Base)                   (Add Base + Size Adjust)
                      |                                       |
                      +-------------------+-----------------------------------------+
                                          |                                         |
                                          v                                         v
                             [ Read Addon Item Cost ]                    [ Final Invoice Cost ]
                             - Add addon cost adjustments
```

### Inheritance Calculation Pipeline:
- **Condition A (Base Item Only):** When a product is ordered without any size modifications or variants, its price is equal to the product's `base_price`.
- **Condition B (Size Override):** When a specific size modifier is selected, the item cost is calculated as the product's base price plus the size's price adjustment:
  $$\text{Final Unit Cost} = \text{Product Base Price} + \text{Size Price Adjustment}$$
- **Condition C (Variant Override):** If the product uses inventory variants (such as a specific size/option combination), the item cost is set directly to the variant's absolute price, overriding the product's base price.
- **Condition D (Add-on Calculations):** For any selected add-on customization items, their individual costs are added to the final unit price:
  $$\text{Final Item Cost} = \text{Base calculated cost} + \sum (\text{Add-on Option prices})$$

## 4.4 Comprehensive Order Lifecycle State Machine
Orders transition through a defined set of states to ensure consistent processing and accurate tracking across the system.

```
       [ DRAFT ] 
           |
           v
      [ PENDING ]  ------( Payment Fails )-------> [ CANCELLED ]
           |
           v ( Payment Success )
      [ ACCEPTED ] 
           |
           v ( Kitchen Admits Ticket )
     [ PREPARING ] 
           |
           v ( Cooking Done )
       [ READY ] 
           |
           +-----------------------+-----------------------+
           |                       |                       |
           v ( Dine-in Serve )     v ( Takeaway Picked )   v ( Delivery Done )
     [ COMPLETED ]           [ COMPLETED ]           [ COMPLETED ]
```

### State Transitions & System Actions:
1. **DRAFT:** The customer constructs their cart. No database transaction is initialized yet.
2. **PENDING:** The order is submitted to the database. The system calculates taxes, updates the status to `'PENDING'`, and holds the order until payment is validated.
3. **ACCEPTED:** Payment is processed successfully. The order status updates to `'ACCEPTED'`, a kitchen queue card is generated, and a WebSocket event is broadcast to the branch's KDS displays.
4. **PREPARING:** Kitchen staff accept the ticket on a KDS screen. The status updates to `'PREPARING'`, starting the preparation timer.
5. **READY:** Cooking is completed. The status updates to `'READY'`, a notification is sent to the customer (via SMS or push notification), and the cashier terminal is alerted.
6. **COMPLETED:** The order is served, picked up, or delivered. The status updates to `'COMPLETED'`, final accounting invoice records are generated, and loyalty points are credited to the customer's account.
7. **CANCELLED:** If payment fails or a manager cancels the order, the status updates to `'CANCELLED'`. Any holds on inventory are released and returned to stock.

## 4.5 Real-Time Kitchen Dispatching (KDS) & Queue Management
The Kitchen Display System (KDS) manages kitchen operations and ticket flow in real-time.

- **WebSocket Synchronization:** The KDS communicates with the backend via high-speed, secure WebSockets (Socket.io). 
- **Room Scoping Isolation:** When a KDS terminal connects, it is placed in a WebSocket room scoped to its specific branch: `tenant_id:branch_id:kds`.
- **Dynamic Action Dispatching:** When an order is accepted, the order engine broadcasts a `ticket.created` event containing the order's items, options, and special notes to the branch's KDS room, updating KDS screens instantly.
- **Priority Escalation Logic:** The system monitors preparation times. If a ticket exceeds the product's estimated preparation time, its priority is escalated to `'RUSH'` on KDS screens, changing its border color to alert kitchen staff.

## 4.6 Cryptographically Secure QR Ordering Workflow
To prevent fraudulent ordering, table-specific QR codes are protected by dynamic cryptographic signatures.

- **Dynamic Token Generation:** Each table record contains a unique `qr_code_token` generated using a secure SHA-256 HMAC hash that signs the combination of the `tenant_id`, `branch_id`, `table_number`, and a rotating system pepper key:
  $$\text{Token} = \text{HMAC-SHA256}(\text{TenantID} + \text{BranchID} + \text{TableNum}, \text{SystemPepper})$$
- **Request Verification:** When a customer scans a table's QR code and submits an order, the backend validates the table ID and verifies that the provided token matches the database records. Orders with mismatched or missing table tokens are rejected. This prevents users from placing orders for random tables from outside the venue.

## 4.7 Subscription Gating & Usage-Based Billing Cycles
The system enforces subscription plan limits dynamically on every administrative request.

- **Middleware Enforcement:** Before executing mutating actions in the backoffice (such as adding branches or products), authorization middleware queries the tenant's current subscription status and plan limits.
- **Action Blocking:** If the tenant has reached their plan's limits, the middleware blocks the operation and returns an HTTP 403 Forbidden response with a descriptive error message (e.g., `'Branch limit reached. Please upgrade to a higher tier.'`).
- **Billing Sync Automation:** Subscriptions and cancellations are kept in sync via Stripe webhooks. When a webhook event is received, the billing service updates the tenant's subscription status in the database, updating their access permissions instantly.

## 4.8 Role-Based Permission Matrix Execution Path
Access control permissions are checked sequentially on every request to protect system resources.

1. **Access Token Extraction:** The gateway extracts the bearer JWT token from the incoming request's `Authorization` header.
2. **Signature Verification:** The gateway decrypts and verifies the token's cryptographic signature using the system's public key.
3. **Payload Verification:** The gateway parses the token's payload to extract user identity attributes, active roles, and assigned permissions.
4. **Tenant Scope Verification:** The system verifies that the target resource's `tenant_id` matches the user's tenant context.
5. **Guard Enforcement:** A custom NestJS permission guard evaluates the controller's required permissions against the user's active permissions. If the user lacks the necessary permissions, the request is rejected with an HTTP 403 Forbidden response and logged in the audit logging pipeline.

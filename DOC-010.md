# 9. PERFORMANCE OPTIMIZATION & HORIZONTAL SCALING

## 9.1 Redis Cache-Aside & Write-Through Caching Topologies
The platform uses high-speed Redis caches to minimize database load and speed up response times:
- **Cache-Aside Pattern:** High-read, low-write assets (such as menu items and categories) are cached using a Cache-Aside pattern:
  - When a client requests a menu, the application checks for the cached menu in Redis first.
  - If the menu is cached, it is returned immediately.
  - If there is a cache miss, the application queries the PostgreSQL database, writes the result to Redis, and returns the menu to the client.
- **Cache Invalidation Policies:** When a restaurant manager modifies their menu, the menu service invalidates the cached menu in Redis instantly. To prevent stale data, all cached keys are configured with a default time-to-live expiration (exactly 2 hours).

---

## 9.2 Advanced Database Optimization (Query Plans, Vacuuming, Index Maintenance)
To maintain quick query speeds as database tables grow, the system applies continuous database tuning:
- **Query Plan Profiling:** Database migrations and complex query filters are analyzed using PostgreSQL `EXPLAIN ANALYZE` tools to identify slow queries and optimize index design.
- **Automated Vacuuming:** Postgres `autovacuum` parameters are configured to regularly clean up dead rows, prevent table bloat, and update table statistics.
- **Database Connection Pooling:** **PgBouncer** connection pools are deployed in front of database replicas, maximizing concurrent connection limits and reducing connection initialization overhead.

---

## 9.3 Client-Side Performance & Bundle Optimizations
To ensure fast load times, particularly on mobile devices scanning QR codes, frontend applications apply modern performance optimizations:
- **Next.js Image Component:** Dynamic images are loaded using the Next.js `<Image />` component. Next.js optimizes images dynamically, converting them to WebP format, resizing them, and applying lazy loading automatically to speed up page loads.
- **Dynamic Imports & Code Splitting:** Heavy administrative modules and non-critical components are loaded dynamically on-demand using Next.js `dynamic()` imports, reducing the initial JavaScript bundle size.

---

## 9.4 Asynchronous Worker Architecture (BullMQ, Redis Streams)
The system uses a distributed background worker pool to handle non-blocking, resource-intensive tasks:
- **Distributed Queues:** Background processes (such as email dispatches, SMS notifications, image optimization, and webhook deliveries) are queued and processed asynchronously using **BullMQ** on a Redis database.
- **Failure Recovery:** Failed jobs are automatically retried up to 3 times using exponential backoff. If a job repeatedly fails, it is moved to a Dead Letter Queue (DLQ) for analysis and debugging.

---

## 9.5 Kubernetes-Driven Horizontal Pod Autoscaling (HPA)
The application container cluster scales dynamically to handle sudden spikes in traffic:
- **Horizontal Pod Autoscaling:** Container clusters are managed using Kubernetes or AWS ECS on Fargate. Dynamic autoscaling metrics are configured to scale the cluster out or in automatically based on application resource usage:
  - Scaling Out: The cluster launches additional container instances when average CPU usage exceeds 70% or average memory usage exceeds 80%.
  - Scaling In: Once usage falls below the thresholds, unnecessary containers are safely terminated to optimize infrastructure costs.

```typescript
'use client';

import React, { useState, useMemo } from 'react';
import Image from 'next/image';

interface ProductSize {
  id: string;
  name: string;
  priceAdjustment: number;
}

interface ProductVariant {
  id: string;
  name: string;
  price: number;
  stockQuantity: number;
}

interface AddonItem {
  id: string;
  name: string;
  price: number;
  isAvailable: boolean;
}

interface ProductAddonGroup {
  id: string;
  name: string;
  minSelections: number;
  maxSelections: number;
  options: AddonItem[];
}

interface Product {
  id: string;
  name: string;
  description: string;
  imageUrl: string | null;
  basePrice: number;
  calories: number | null;
  isAvailable: boolean;
  sizes: ProductSize[];
  variants: ProductVariant[];
  addons: ProductAddonGroup[];
}

interface Category {
  id: string;
  name: string;
  products: Product[];
}

interface MenuBrowserProps {
  categories: Category[];
  primaryColor: string;
}

export const MenuBrowser: React.FC<MenuBrowserProps> = ({ categories, primaryColor }) => {
  const [searchQuery, setSearchQuery] = useState('');
  const [selectedCategoryId, setSelectedCategoryId] = useState<string>('all');
  const [activeCartProduct, setActiveCartProduct] = useState<Product | null>(null);

  // Cart Configuration States
  const [selectedSize, setSelectedSize] = useState<ProductSize | null>(null);
  const [selectedVariant, setSelectedVariant] = useState<ProductVariant | null>(null);
  const [selectedAddons, setSelectedAddons] = useState<AddonItem[]>([]);
  const [quantity, setQuantity] = useState(1);

  // Filter Categories & Products dynamically
  const filteredCategories = useMemo(() => {
    return categories
      .map((category) => {
        const matchingProducts = category.products.filter((product) => {
          const matchesSearch = product.name.toLowerCase().includes(searchQuery.toLowerCase()) ||
            product.description.toLowerCase().includes(searchQuery.toLowerCase());
          return product.isAvailable && matchesSearch;
        });

        return { ...category, products: matchingProducts };
      })
      .filter((category) => {
        const matchesCategorySelection = selectedCategoryId === 'all' || category.id === selectedCategoryId;
        return matchesCategorySelection && category.products.length > 0;
      });
  }, [categories, searchQuery, selectedCategoryId]);

  // Dynamic Inheritance Pricing Logic
  const calculatedUnitPrice = useMemo(() => {
    if (!activeCartProduct) return 0;
    
    let base = Number(activeCartProduct.basePrice);

    // Apply Size Adjustment (Condition B)
    if (selectedSize) {
      base += Number(selectedSize.priceAdjustment);
    }

    // Apply Variant Absolute Override (Condition C)
    if (selectedVariant) {
      base = Number(selectedVariant.price);
    }

    // Add selected customizations (Condition D)
    const addonsTotal = selectedAddons.reduce((sum, addon) => sum + Number(addon.price), 0);

    return base + addonsTotal;
  }, [activeCartProduct, selectedSize, selectedVariant, selectedAddons]);

      const activeGroupSelections = selectedAddons.filter((item) => 
        group.options.some((opt) => opt.id === item.id)
      );

      if (activeGroupSelections.length < group.maxSelections) {
        setSelectedAddons([...selectedAddons, addon]);
      } else if (group.maxSelections === 1 && activeGroupSelections.length === 1) {
        // Auto-replace single-choice selections
        const groupOptionIds = group.options.map((opt) => opt.id);
        const filteredAddons = selectedAddons.filter((item) => !groupOptionIds.includes(item.id));
        setSelectedAddons([...filteredAddons, addon]);
      }
    }
  };

  const resetCartModal = () => {
    setActiveCartProduct(null);
    setSelectedSize(null);
    setSelectedVariant(null);
    setSelectedAddons([]);
    setQuantity(1);
  };

  return (
    <div className="w-full max-w-md mx-auto bg-gray-50 min-h-screen pb-24">
      {/* Dynamic Brand Navigation Bar */}
      <header className="sticky top-0 bg-white shadow-sm z-30 px-4 py-3">
        <input
          type="text"
          placeholder="Search menu items..."
          className="w-full px-4 py-2 border rounded-full text-sm bg-gray-100 focus:outline-none focus:ring-2 focus:ring-offset-2"
          style={{ '--tw-ring-color': primaryColor } as React.CSSProperties}
          value={searchQuery}
          onChange={(e) => setSearchQuery(targetQuery => e.target.value)}
        />
        <div className="flex gap-2 overflow-x-auto py-2 scrollbar-none mt-2">
          <button
            onClick={() => setSelectedCategoryId('all')}
            className={`px-3 py-1 rounded-full text-xs font-semibold whitespace-nowrap transition-colors ${
              selectedCategoryId === 'all' ? 'text-white' : 'bg-gray-100 text-gray-700'
            }`}
            style={selectedCategoryId === 'all' ? { backgroundColor: primaryColor } : {}}
          >
            All
          </button>
          {categories.map((category) => (
            <button
              key={category.id}
              onClick={() => setSelectedCategoryId(category.id)}
              className={`px-3 py-1 rounded-full text-xs font-semibold whitespace-nowrap transition-colors ${
                selectedCategoryId === category.id ? 'text-white' : 'bg-gray-100 text-gray-700'
              }`}
              style={selectedCategoryId === category.id ? { backgroundColor: primaryColor } : {}}
            >
              {category.name}
            </button>
          ))}
        </div>
      </header>

      {/* Structured Category Lists */}
      <main className="px-4 py-4 space-y-8">
        {filteredCategories.map((category) => (
          <section key={category.id}>
            <h2 className="text-lg font-bold text-gray-900 border-l-4 pl-2 mb-4" style={{ borderColor: primaryColor }}>
              {category.name}
            </h2>
            <div className="grid grid-cols-1 gap-4">
              {category.products.map((product) => (
                <div
                  key={product.id}
                  onClick={() => setActiveCartProduct(product)}
                  className="bg-white p-3 rounded-xl flex shadow-sm cursor-pointer hover:shadow-md transition-shadow"
                >
                  <div className="flex-1 pr-3">
                    <h3 className="font-semibold text-gray-900 text-sm">{product.name}</h3>
                    <p className="text-gray-500 text-xs mt-1 line-clamp-2">{product.description}</p>
                    <span className="text-gray-900 font-bold text-sm block mt-2">
                      ${Number(product.basePrice).toFixed(2)}
                    </span>
                  </div>
                  {product.imageUrl && (
                    <div className="relative w-20 h-20 rounded-lg overflow-hidden flex-shrink-0">
                      <Image
                        src={product.imageUrl}
                        alt={product.name}
                        fill
                        sizes="80px"
                        className="object-cover"
                        loading="lazy"
                      />
                    </div>
                  )}
                </div>
              ))}
            </div>
          </section>
        ))}
      </main>

      {/* Item Customization Drawer / Modal Container */}
      {activeCartProduct && (
```

## C.2 Kitchen Display System (KDS) Queue Terminal Screen

The following React Component integrates directly with Socket.io real-time broadcast servers, receiving KDS ticketing signals and performing interactive priority ticket status rendering.

```typescript
'use client';

import React, { useEffect, useState } from 'react';
import io from 'socket.io-client';

interface OrderItem {
  id: string;
  name: string;
  quantity: number;
  size: string | null;
  addons: string[];
  cookingStatus: 'PENDING' | 'PREPARING' | 'COOKED';
}

interface KDSTicket {
  ticketId: string;
  ticketNumber: string;
  priority: 'LOW' | 'NORMAL' | 'RUSH';
  elapsedMinutes: number;
  items: OrderItem[];
}

export const KDSTerminal: React.FC<{ branchId: string; accessToken: string }> = ({ branchId, accessToken }) => {
  const [tickets, setTickets] = useState<KDSTicket[]>([]);

  useEffect(() => {
    // Handshake initialization with secure credentials validation
    const socket = io(process.env.NEXT_PUBLIC_SOCKET_SERVER_URL || '', {
      auth: { token: accessToken },
      query: { branchId }
    });

    socket.on('connect', () => {
      console.log('KDS WebSocket established successfully.');
    });

    // Handle new incoming tickets
    socket.on('ticket.created', (newTicket: KDSTicket) => {
      setTickets((prevTickets) => [...prevTickets, { ...newTicket, elapsedMinutes: 0 }]);
    });

    // Update active state mappings
    socket.on('ticket.item_updated', (updatePayload: { orderItemId: string; status: 'PREPARING' | 'COOKED' }) => {
      setTickets((prevTickets) =>
        prevTickets.map((ticket) => ({
          ...ticket,
          items: ticket.items.map((item) =>
            item.id === updatePayload.orderItemId
              ? { ...item, cookingStatus: updatePayload.status }
              : item
          )
        }))
      );
    });

    return () => {
      socket.disconnect();
    };
  }, [branchId, accessToken]);

  const updateItemStatus = async (orderItemId: string, currentStatus: string) => {
    const nextStatus = currentStatus === 'PENDING' ? 'PREPARING' : 'COOKED';
    
    // Optimistic client mutation
    setTickets((prevTickets) =>
      prevTickets.map((ticket) => ({
        ...ticket,
        items: ticket.items.map((item) =>
          item.id === orderItemId ? { ...item, cookingStatus: nextStatus as any } : item
        )
      }))
    );

    await fetch(`/api/v1/kds/items/${orderItemId}/status`, {
      method: 'PUT',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${accessToken}`
      },
      body: JSON.stringify({ status: nextStatus })
    });
  };

  return (
    <div className="w-full bg-slate-900 min-h-screen p-6 flex flex-col">
      <header className="flex justify-between items-center border-b border-slate-800 pb-4 mb-6">
        <h1 className="text-xl font-bold text-white">Active Production Board</h1>
        <div className="flex gap-4">
          <span className="text-xs text-slate-400">Connected</span>
          <span className="w-3 h-3 rounded-full bg-green-500 self-center"></span>
        </div>
      </header>

      {/* Grid of Kitchen Tickets */}
      <main className="flex-1 grid grid-cols-4 gap-6 items-start overflow-x-auto">
        {tickets.map((ticket) => (
          <div
            key={ticket.ticketId}
            className={`rounded-xl border shadow-xl flex flex-col max-h-[80vh] overflow-hidden ${
              ticket.priority === 'RUSH' ? 'border-red-500 bg-red-950/20' : 'border-slate-800 bg-slate-950'
            }`}
          >
            {/* Ticket Header card */}
```

# APPENDIX F: OPERATIONAL RUNBOOKS & TROUBLESHOOTING RUNBOOKS

This appendix provides production-grade operational runbooks, incident recovery steps, and diagnostic procedures for site reliability engineers (SREs) and platform operations teams managing the Zayjar Restaurant SaaS Platform.

## F.1 Database Failover & High-Availability Recovery Plan

In the event of a physical primary database node failure on AWS RDS, the following sequence details the automated failover execution path, proxy connection cycling, and read-replica re-routing.

```
       [ AWS RDS Primary Fail ]
                  |
                  v ( Heartbeat loss detected by Standby Node )
     [ Standby Node Promoted ]
                  |
                  +-----------------------+-----------------------+
                  |                                               |
                  v ( AWS DNS Updates )                           v ( PgBouncer Connection Reset )
        [ Read/Write Resumed ]                          [ Active Queries Flushed ]
```

### Incident Diagnostics & Failover Verification Sequence:
1. **Heartbeat Loss Detection:** The AWS RDS Multi-AZ manager registers heartbeat loss on the Primary database instance (e.g., node timeout or storage volume lockup).
2. **Standby Node Promotion:** The standby RDS node is promoted to become the new Primary Write instance. 
3. **AWS DNS Routing Update:** AWS updates the virtual IP address mapped to the main RDS writer connection endpoint. This update usually propagates within 30 to 60 seconds.
4. **PgBouncer Connection Cycling:** During the database promotion, active database queries on application containers will throw socket timeout exceptions (such as `P2025` or transaction execution timeouts). To clear dead socket connections immediately:
   - SREs or automated scaling agents execute a connection flush command on PgBouncer nodes:
     ```bash
     pgbouncer -u pgbouncer_admin -p SecretPgbouncerAdmin123! -c "RECONNECT"
     ```
   - This command drops all cached sockets and forces PgBouncer to establish new, clean socket connections to the newly promoted RDS writer endpoint.
5. **Secondary Verification checks:** SREs execute verification checks on the core database cluster to ensure read and write operations have resumed successfully:
   ```sql
   -- Verify write operations
   INSERT INTO audit_logs (action, entity_name, ip_address, user_agent)
   VALUES ('FAILOVER_DIAGNOSTIC_WRITE', 'RDS_HEALTH_CHECK', '127.0.0.1', 'AWS_HEALTH_MONITOR');
   
   -- Verify replica replication lag
   SELECT pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS replication_lag_bytes
   FROM pg_stat_replication;
   ```

## F.2 Cache Invalidation Failures & Redis Desynchronization Runbook

In rare operational scenarios where Redis cache entries fall out of sync with the persistent PostgreSQL database (for example, if a background worker fails or a network partition blocks cache invalidation events), customer menus may display stale menu items or incorrect prices.

### Diagnostic Steps & Runbook Remedies:
1. **Locating Cache Desynchronization:** To confirm a cache desynchronization event, SREs compare the target tenant's product properties in PostgreSQL directly against the serialized string stored in Redis:
   ```bash
   # Read cached product object from Redis
   redis-cli -u redis://:SecretRedis123!@redis-cache:6379 GET "tenant:7a18:product:p222"
   ```
   If the Redis JSON output displays outdated base prices compared to the database, a cache invalidation failure has occurred.
2. **Targeted Cache Invalidation:** To restore service immediately without affecting other tenant caches, SREs delete the specific tenant's stale menu cache key in Redis:
   ```bash
   # Remove targeted cache key
   redis-cli -u redis://:SecretRedis123!@redis-cache:6379 DEL "tenant:7a18:menu"
   ```
   Subsequent client requests to the customer QR menu will result in a cache miss, prompting the application server to query PostgreSQL and cache the updated menu data.
3. **Emergency Cache Flush (Cluster-Wide):** If a system-wide database update fails to invalidate caches across the entire network, SREs can execute a cluster-wide cache flush:
   ```bash
   # Flush cached query entries across all cluster nodes safely
   redis-cli -u redis://:SecretRedis123!@redis-cache:6379 FLUSHALL ASYNC
   ```
   *Warning:* Flush operations must use the `ASYNC` modifier to perform the cleanup in the background, preventing Redis from blocking and timing out active WebSocket and queue services.

## F.3 Multi-Tenant Outbound Webhook Queue Backlog Incident Response

If a tenant's third-party integration server goes offline, outbound webhook dispatch queues may build up in Redis, consuming system memory and slowing down background notifications.

### Action Plan & Remediation Sequence:
1. **Identify Backlog Volume:** SREs log into the Redis console and query the size of active and failed BullMQ webhook queues:
   ```bash
   # Check active queue lengths
   redis-cli -u redis://:SecretRedis123!@redis-cache:6379 LLEN "bull:webhooks:active"
   redis-cli -u redis://:SecretRedis123!@redis-cache:6379 LLEN "bull:webhooks:wait"
   ```
2. **Identify Failing Endpoints:** Review Winston logs to pinpoint the specific `tenant_id` and webhook target URL that is causing connection timeouts:
   ```bash
   # Parse application logs for webhook failure warnings
   grep "WEBHOOK_DISPATCH_FAILED" /var/log/zayjar-api/error.log | jq '.metadata' | sort | uniq -c
   ```
3. **Enforce Temporary Webhook Pause:** If a specific tenant's integration server remains offline and causes a backlog of over 50,000 failed webhook requests, SREs temporarily pause the tenant's webhook subscription to protect system resources:
   ```sql
   -- Pause the failing tenant's webhook in PostgreSQL
   UPDATE webhooks 
   SET is_active = FALSE 
   WHERE tenant_id = '7a18f-39b0-4050-bf83-097a18fcd34b' AND target_url = 'https://broken-partner.com/callback';
   ```
4. **Purging Failed Backlog Jobs:** SREs clear out failed webhook jobs from the active queue to restore background processing speeds:
   ```bash
   # Safe background cleanup of waiting job lists
   redis-cli -u redis://:SecretRedis123!@redis-cache:6379 DEL "bull:webhooks:wait"
   ```
   Once the tenant's integration server is back online, the webhook configuration can be reactivated, and subsequent event dispatches will resume automatically.
